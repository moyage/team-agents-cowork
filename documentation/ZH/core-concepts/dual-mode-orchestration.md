# 双模式编排 (Dual-Mode Orchestration)

## 1. 背景与挑战
在构建多智能体系统的实践中，开发者往往面临一个核心两难：
一方面，像 Kimi、Claude 等底层 LLM 能力不断提升，它们能够作为一个自治的“黑盒”出色地完成长链条的开放式任务（例如代码探索、文档概括）；
另一方面，在企业级的软件工程或自动化流水线中，又需要绝对的“确定性”，要求任务的每一步都必须可验证、可追踪且高度受控（即“白盒”或“编排”要求）。
传统的框架要么强行要求所有任务编排，导致灵活性丧失；要么完全依赖单一 Agent 黑盒盲跑，导致失败率不可控。基于此，Team Agents Cowork 提出了**双模式编排**架构。

## 2. 场景映射：Kimi-Code vs OpenCode
我们可以通过对比业界常用的两种形态来理解双模式：
- **Kimi-Code 范式 (黑盒独立模式)**：将整个大目标抛给模型，它自主规划、自主调用终端/文件读取等工具，中间过程对系统是一个黑盒，最终给出结果。这种方式探索性强、开销小。
- **OpenCode/Devin 范式 (白盒编排模式)**：整个过程被严格切分为需求分析、架构设计、编码、测试、Review等环节。每个环节由专职智能体把控，并被高层系统统筹编排。这种方式确定性高、易排错。

## 3. 核心定义
**双模式编排 (Dual-Mode Orchestration)** 指的是在 L3 层（团队编排层），系统原生支持并能动态选择两种任务执行范式：
- **黑盒独立模式 (Solo/Blackbox)**
- **编排团队模式 (Team/Orchestrated)**

## 4. 特性对比

### 4.1 黑盒独立模式 (Solo Mode)
- **执行主体**: 单个具备高自治能力的高级 L2 节点智能体。
- **流程控制**: 智能体内部自主决定思维链 (CoT) 和工具调用顺序。
- **配置开销**: 极低，仅需一条 prompt 目标即可运行。
- **容错与回滚**: 依赖智能体自身的内部重试机制。
- **适用场景**: 数据清洗、探索性数据分析 (EDA)、单模块代码重构、开放式问答。

### 4.2 编排团队模式 (Team Mode)
- **执行主体**: 由 L3 引擎统一调度的多个 L2 智能体集群。
- **流程控制**: 严格按照预定义的 YAML DAG (有向无环图) 节点依赖进行流转。
- **配置开销**: 较高，需要定义节点、边界、输入输出协议。
- **容错与回滚**: 支持节点级别的故障隔离与重启，失败不影响整个图结构崩溃。
- **适用场景**: 复杂的长链路软件开发流水线、带有审核机制的自动化发布、多源信息交叉验证。

## 5. 执行示例

**触发 Solo 模式的命令：**
```bash
# 智能体将自行决定如何完成这个模糊的任务
npm run start:solo -- --task="帮我分析当前项目的架构瓶颈并出具报告"
```

**触发 Team 模式的命令：**
```bash
# 系统将解析 DAG，按拓扑顺序调度架构师、开发者和审核专家
npm run start:team -- --workflow=examples/advanced-pipeline-dag.yaml
```

## 6. 总结
双模式编排是 Team Agents Cowork “驾驭工程 (Harness Engineering)” 的最佳体现。系统不再与底层模型的能力相抗衡，而是提供灵活的网关：在模型擅长的领域释放其自主性 (Solo)，在需要工程严谨性的领域施加强力管束 (Team)。
