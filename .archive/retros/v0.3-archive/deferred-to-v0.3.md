# Deferred to V0.3.0 (and V0.3 Scope Boundary)

本文档记录在 V0.2.0 开发与发布过程中被识别并延后（Deferred）的事项。
基于最新的产品决策，本文档目前同时承担 **“V0.3 deferred scope 整理”** 与 **“V0.3 planning boundary clarification（规划边界澄清）”** 的作用。

---

## 🔴 V0.3 当前迭代必须纳入的能力范围 (Committed for V0.3)
以下主题在最初可能仅作为模糊的远期备忘，但基于最新的评估，**它们已全部被提权并明确划入 V0.3 的核心交付范围**。

### 1. 中央化远端分发 (Remote Installation Baseline)
- **问题/机会**：当前用户必须 `git clone` 整个仓库再执行 `./setup.sh`，这对于只是想给当前项目快速注入协议的开发者来说太重了。
- **为何是 V0.3 必做**：扫除推广期的最大摩擦力，从“下载源码”升级为“一条命令注入”。
- **预期价值**：将 3 分钟的接入时间压缩到 30 秒内（一键执行）。
- **风险/依赖**：需重构脚本结构以支持 `curl -sL ... | bash` 的无状态网络流执行。

### 2. 自动环境嗅探 (Auto-sniffing IDE Target)
- **问题/机会**：目前依赖交互输入或传参 `--ide`。对于多工具混用的团队容易输错。
- **为何是 V0.3 必做**：远端分发往往是静默执行的，必须配合自动环境嗅探才能做到真正的“零配置”。
- **预期价值**：自动检测 `.cursor/`、`.trae/` 或 `.vscode/` 目录，无感下发对应的适配器文件。

### 3. 反馈收集与需求池机制
- **问题/机会**：团队内各个角色在使用协议时，需要一种标准的路径来报告痛点或提出新增规范的诉求。
- **为何是 V0.3 必做**：V0.2 开启了试点的可能性，V0.3 就必须接住试点过程中的反馈。
- **预期价值**：建立标准化的 Issues 模板或基于 Markdown 的需求收口池，形成体验闭环。

### 4. 流程中途上游返修支持
- **问题/机会**：现实开发中经常发生“Build 阶段发现 Spec 有误”的情况，目前协议缺乏对“中途向上游打回”的规范。
- **为何是 V0.3 必做**：这是填补理想化瀑布流到真实工程环境落差的关键一步。
- **预期价值**：允许并规范化从 Build 阶段逆向回滚到 Plan 甚至 Spec 阶段的协同动作。

### 5. 多轮迭代项目演进支持
- **问题/机会**：协议此前的视角多为“从零到一做完就结束”。如何进行 v1.0 到 v2.0 的安全演进？
- **为何是 V0.3 必做**：生命周期管理的必备能力。**注：本次从 V0.2 发布后重组规划、继承 Backlog、收口遗留项并启动 V0.3 的整个过程，本身就被作为该能力的第一轮真实实践样本。**
- **预期价值**：形成跨版本 Backlog 继承、旧版本状态收敛、多轮发布的标准化 Markdown 协作拓扑。

### 6. 防篡改 CI
- **问题/机会**：缺乏物理验证手段阻止协议在端侧被开发者私自抹除或修改。
- **为何是 V0.3 必做**：一旦推广，需要刚性约束来防止架构漂移。
- **预期价值**：通过 GitHub Actions 对注入的 `.cursorrules` 等文件进行 Hash 比对，防止私自绕过。

## 🔵 降级到 Later / Post-V0.3 的方向
以下主题在当前阶段因优先级不高或实现过于庞大，明确不纳入 V0.3 范畴，保留为后续演进选项：

### 1. 多 repo 同步中心
- **问题/描述**：支持大型组织级别的 Profile 统一下发，以及在多个不同微服务仓库之间保持协议文件同步。
- **为何降级 Later**：这超出了“独立开发者与小微团队”的 V0.3 目标受众，属于企业级的庞大工程。当前优先夯实单仓体验。

### 2. 更深诊断能力
- **问题/描述**：在 Agent IDE 内部署深度的探针钩子（Advanced Doctor Probes），收集每次调用的质量数据。
- **为何降级 Later**：极易触及隐私边界且依赖特定 IDE 的私有 API，维护成本极高，违背协议层的轻量级初衷。
