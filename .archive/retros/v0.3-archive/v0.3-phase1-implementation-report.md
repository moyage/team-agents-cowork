## 4. 远端分发采用了什么实现方案
**平滑升迁方案**（在 `install.sh` 中实现）：
通过检测运行时目录内是否存在 `setup.sh` 来判断执行环境：
- 若在本地 clone 的仓库中执行，直接透明透传参数调用 `./setup.sh`。
- 若处于远端模式（例如通过 `curl ... | bash` 直接执行），则：
  1. 在本地使用 `mktemp` 创建一个随机临时目录（如 `/tmp/team-agents-cowork-XXXXXX`）。
  2. 使用 `trap` 注册退出时的目录自动销毁机制，保证宿主机“用后无痕”。
  3. 执行 `git clone --depth 1` 从 `main` 分支拉取最简仓库，并切换至临时目录执行带有全部下传参数的 `setup.sh`。

## 5. 自动环境嗅探采用了什么判定逻辑
**目标目录特征检测机制**（在 `setup.sh` 中实现）：
嗅探发生在获得目标目录（`TARGET_DIR`）之后：
- 优先级判定链：
  - 如果目标目录存在 `.cursor` 文件夹或 `.cursorrules`，则识别为 `cursor`。
  - 如果目标目录存在 `.trae` 文件夹或 `.traerules`，则识别为 `trae`。
  - 如果目标目录存在 `CLAUDE.md`，则识别为 `claude_code`。
  - 作为兜底，如果目标目录仅存在 `.vscode` 且无其他特征，默认退让至使用度最广的 `cursor`。
- 用户显式传入参数的优先级高于自动嗅探结果。
- 当且仅当无法嗅探出特征时，才会安全降级到原有的交互式 `read -p` 用户提问环节（且支持 CI 下的快速失败退出）。
