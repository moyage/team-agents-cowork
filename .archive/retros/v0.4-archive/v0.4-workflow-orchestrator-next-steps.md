# V0.4 Workflow Orchestrator Next Steps

基于评估结论，V0.4 的落地执行应采取“分阶段、双线并行”的策略，以确保不破坏现存 V0.3 的稳定性。

## 1. 建议的下一步动作 (Next Actions)

### 阶段 1: 个人域 MVP (MCP 集成)
* **前置条件**: Cursor 或 Claude Code 环境准备。
* **执行任务**: 将 `scripts/verify_integrity.py` 和 `scripts/build_demand_pool.py` 包装为一个极其简单的本地 MCP Server。
* **预期收益**: 证明 Agent 可以自主发现协议是否被篡改，并自主提取精简后的需求池上下文，而无需人类反复复制粘贴。

### 阶段 2: 团队协作 MVP (GitHub Actions 触发)
* **前置条件**: V0.3 的完整性校验流水线（已就绪）。
* **执行任务**: 新增一个 Actions Workflow，当人类提交含有 `templates/UPSTREAM_REVISION.md` 的 PR 时，自动唤起一个受限的审查 Agent（如基于轻量 API 脚本）对其影响范围进行校验，并在 PR 评论中输出“是否允许状态机向下跃迁”的结论。
* **预期收益**: 证明防线可以物理拦截并自动流转。

## 2. 不建议立即做的事项 (Out of Scope for V0.4 MVP)
* ❌ **引入沉重的 Python 状态机引擎 (如 LangGraph/Temporal)**：这会直接导致本协议失去“轻量化”和“纯静态文本”优势，把普通开发者拒之门外。
* ❌ **让 Agent 获得 `workflow/iteration-state.json` 的自主写入权限**：在当前幻觉概率下，核心状态机的跃迁（如版本完结）仍必须留给人类或硬编码的安全 CI 脚本控制。

## 3. 前置条件与依赖
目前 V0.3 的底座非常健康，`demand-pool` 的构建能力也已初具雏形。为了更好地供给 MVP，还需要进行一项最小改进：
* 优化 `demand-pool` 结构，将其变更为易被机器序列化读取的 JSON 或强结构化 Markdown，以减少 MCP 读取时的 Token 浪费。
