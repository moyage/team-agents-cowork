# V0.4 初始需求池 (Initial Demand Pool)

*来源汇总：V0.3 遗留项 (Deferred)、验收审计反馈、多视角复盘 (Multi-Perspective Review)*

## 1. 自动推进与工作流编排 (P0 - Must)
* **来源**: 架构视角复盘、长期演进路线。
* **描述**: 引入 Workflow Orchestrator Evaluation。当前协议主要提供“静态约束与文档模板”，下一步需研究如何使用自动化平台 (如 GitHub Actions/MCP) 驱动 Planner、Coder、Reviewer 间基于模板自动交接。
* **影响范围**: 全局架构。

## 2. 需求池压缩与结构化 (P1 - Should)
* **来源**: 扩展性复盘。
* **描述**: 重构 `scripts/build_demand_pool.py`，目前是纯文本堆叠。当反馈过多时会污染 AI Context。需增加文本去重、高频聚类能力，或序列化为 JSON 供检索，而不是单体大文件。
* **影响范围**: 协作与维护流程。

## 3. 防篡改基线目标自动发现 (P2 - Could)
* **来源**: 维护者视角复盘。
* **描述**: `update_integrity_manifest.py` 现在需要手动在字典中维护被监控的文件路径。修改为基于模式匹配 (如 `adapters/*.tpl`, `profiles/*.yaml`) 自动发现目标，降低新增规则时的配置成本。
* **影响范围**: 协议规则维护体验。

## 4. 多仓库同步 (Multi-repo sync) (P2 - Could)
* **来源**: V0.3 Deferred 遗留项。
* **描述**: 目前网络分发是“单次覆盖”。如果主协议更新，已分发的项目无法感知。需探索一种轻量级的 `check-updates` 机制或 `sync-pull` 脚本，允许已注入协议的 Repo 定期对齐 V0.x 基线。
* **影响范围**: 跨项目管理。

## 5. 深度诊断工具 (Deep diagnosis) (P3 - Could)
* **来源**: V0.3 Deferred 遗留项。
* **描述**: 为排查由于 `mktemp` 创建临时文件夹引发的边缘错误提供独立的 `scripts/doctor.py` 命令。
* **影响范围**: 极端错误排查 (Troubleshooting)。
