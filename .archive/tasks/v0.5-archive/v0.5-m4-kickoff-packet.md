# V0.5 M4 Kickoff & Reconnaissance Note
## Integration-Focused Validation

### 1. Analysis of M3 Context API Usage
The newly introduced MCP tools provide a secure, protocol-level abstraction for the L2/L3 orchestration layer:
- `get_dispatch_state` (0 arguments): 
  - **Caller:** Orchestrator / Planner Agent.
  - **Usage:** Replaces raw file reads (`cat workflow/dispatch.json`). Safely retrieves the canonical Source-of-Truth (active gate, task, blockers) to determine the next immediate action upon session resume.
- `get_execution_diff` (`baseline_commit` required):
  - **Caller:** Independent Reviewer Sub-Agent (Gatekeeper).
  - **Usage:** Replaces error-prone shell commands (`git diff`). Provides an immutable, semantic payload of execution changes, ensuring the Reviewer assesses the raw truth of the implementation delta without being misled by implementation artifacts.

### 2. Combinatorial Workflow (Orchestrator + Reviewer)
1. **Orchestrator** initializes and calls `get_dispatch_state`.
2. It detects a `current_gate: mX_result_review_pending`.
3. Orchestrator delegates a task to the **Independent Reviewer**, granting it access to `get_execution_diff`.
4. Reviewer queries the diff, validates it against the execution contract, and produces a physical `decision.json` artifact.
5. Orchestrator resumes, reads the decision, and updates the workflow state manually.

### 3. Minimal Viable M4 Target (MVT)
**Target:** A programmatic integration test client for the MCP Context API.
- **Goal:** Build a read-only script (e.g., `src/mcp-server/src/test-client.ts`) that connects to the MCP server via stdio, calls both `get_dispatch_state` and `get_execution_diff`, and asserts their outputs.
- **Why:** This proves end-to-end integration viability over the actual MCP transport layer, validating the M3 tools without introducing new governance policies or L2 mutators.

### 4. M4 Forbidden Zones (Out of Scope)
- **Workflow Mutation:** No MCP tools that write to `dispatch.json` or `iteration-state.json`.
- **Approval Automation:** No logic that automatically signs off on reviews. The physical generation of `decision.json` by an isolated LLM instance must remain mandatory.
- **Reviewer Identity Redesign:** Do not change the Sub-Agent delegation strategy established in M2/M3.
- **Protocol Overhaul:** Maintain the existing L0-L6 architecture constraints.
