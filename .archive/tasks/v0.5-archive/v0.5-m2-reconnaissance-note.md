# V0.5 M2 Planning Reconnaissance Note

## 1. Current Technical Baseline Summary
The M1 execution successfully established a canonical Native TypeScript MCP Server.
- **Scaffold:** Clean ESM NodeNext project in `src/mcp-server/`.
- **SDK Integration:** Extends `@modelcontextprotocol/sdk` utilizing the `StdioServerTransport`.
- **Probes:** Features a single read-only tool (`get_iteration_state`) that accurately fetches the canonical `workflow/iteration-state.json` using physical `fs/promises`.
- **Validation Chain:** Basic `npm run build` and `tsc --noEmit` are intact and reliable.
- **State Check:** M1 successfully closed under an `Exception_Approved` status, exposing a critical flaw in JSON-based self-reporting (the `.gitignore` architecture drift).

## 2. Candidate M2 Directions

### Direction A: Workflow State Mutators (Write/Progression API)
- **Goal:** Introduce tools like `update_dispatch_gate` or `submit_review_decision` to allow agents to progress the workflow purely via MCP.
- **Value:** Reduces reliance on raw CLI `jq` manipulation.
- **Risk:** High. Giving write access without native validation could allow an agent to illegally advance the state machine.

### Direction B: Protocol Validation & Compliance API (Physical Truth Enforcer)
- **Goal:** Expose MCP tools that perform hard validation. For example, `validate_schema` (migrating Python validation to TS Zod) and, crucially, `verify_execution_compliance` (invoking `git diff --name-only` to cross-reference against `allowed_files` in the contract).
- **Value:** Directly addresses the M1 Retro incident by empowering any Reviewer Agent with a hard API to check for Agentic Sycophancy.
- **Risk:** Low/Medium. Requires safely spawning Git child processes within the MCP server.

### Direction C: Orchestration Agent Toolset
- **Goal:** Expose tools to list active tasks, read demand pools, and fetch open execution contracts.
- **Value:** Helps the planning agent orient itself.
- **Risk:** Low. Just more read-only probes.

## 3. Recommended M2 Focus
**Recommendation: Direction B (Protocol Validation & Compliance API).**

*Rationale:* The M1 Post-Review Compliance Incident proved that an executor's `Result` JSON cannot be trusted. Before we grant the MCP server "Write" capabilities (Direction A) to advance the workflow, the server *must* first serve as the Protocol Enforcer. Building a `verify_execution_compliance` tool natively into the server allows an external Reviewer Agent to securely ask the server: "Did this executor modify files outside the contract?"

**Minimum Verifiable Boundary:** 
1. Implement a tool that checks an artifact against its JSON schema.
2. Implement a tool that runs `git diff --name-only <baseline_commit>` and asserts it against `allowed_files` in a given `execution-contract.json`.

## 4. Separation of Concerns (What remains OUT of scope for M2)
- **Protocol Evolution:** How we physically deploy the *Reviewer Agent Daemon* is out of scope. The MCP server provides the *tools* for the Reviewer, but M2 does not concern itself with orchestrating the Reviewer identity.
- **Concurrency/State Sharding:** Handling simultaneous PRs/branches is out of scope. We remain in a single-lane dispatch model for now.
- **Replacing Python completely:** Legacy `scripts/validate_workflow_state.py` remains as a CI fallback. M2 adds TS capability but doesn't delete the old yet.

## 5. Governance & Compliance Integration (M1 Lessons)
M2 planning **must** account for the Physical Diff Verification rule. The `verify_execution_compliance` tool will be the technical embodiment of this rule, ensuring that any future Review Decision requires calling this tool before emitting an "Approved" artifact.
