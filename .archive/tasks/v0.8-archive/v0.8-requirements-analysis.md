# V0.8 Requirements Analysis

## 1. Goal
Translate the V0.7 theoretical models into a functional engine, specifically addressing the orchestration, participant registration, and workspace topology questions raised in the V0.8 Planning Q&A.

## 2. Participant Registration (The Heterogeneous Collaboration Engine)
- **The Problem:** The framework assumes an implicit, static agent structure (e.g., Hermes executes, `opencode run` reviews). This violates the "Low Invasiveness" principle.
- **The Solution:** A Dynamic, Capability-Based Registry (`.agent-state/registry.json`).
  - **Action:** Agents (Hermes, Opencode, SubAgents) must dynamically register their capabilities (`supports_mcp`, `is_isolated_reviewer`) when initialized.
  - **Orchestration:** The Main Brain (Coordinator) must assign sub-tasks (Contracts) based on required capabilities, not hardcoded names.

## 3. Team Mode Orchestration & Workspace Topology
- **The Problem:** A centralized "Main Brain" cannot orchestrate the local environments of remote developers.
- **The Solution:** A Decentralized, Dual-Tiered Sync Topology (Local Bus + Cloud Ledger).
  - **Local Bus (`.agent-state/`):** The collaborative workspace within the target project's local directory. Hermes, Opencode, and the Reviewer SubAgents read/write to this local bus. It handles high-frequency, trial-and-error state transitions (`contract_review_pending` -> `execution_ready`).
  - **Cloud Ledger (GitHub):** The remote repository acts as the Source of Truth for *Milestone-Level* and *Gate-Level* transitions. Only formally approved state transitions (e.g., an Approved `execution-result.json` accompanied by the actual code commit) are pushed to the Cloud Ledger (e.g., a dedicated `agent-sync` branch).
  - **The "Pull" Model:** Remote team members (and their respective local AI tools) do not "register" centrally. They clone the repository, pull the `.agent-state/` updates, and *claim* `PENDING` contracts by updating the assignee field locally and pushing the state change.

## 4. V0.7 Carry-over Roadmaps
- **`.agent-state/` Physical Migration:** Refactoring the MCP Server and bootstrap scripts to operate statelessly against the target repository's `.agent-state/` directory while preserving V0.7 audit logs in the framework's own `tasks/` directory.
- **SubAgent Review Automation Daemon:** Creating a background hook (`agent-protocol-review`) that autonomously invokes the isolated Gatekeeper (e.g., `opencode run`) upon detecting a `_review_pending` state transition in `.agent-state/workflow/dispatch.json`.
- **Legacy Tooling Deprecation:** Injecting deprecation warnings into the Python scripts (`scripts/*.py`) as defined in the V0.7-M5-B roadmap.
