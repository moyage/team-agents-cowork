# V0.8 Scope Definition

The scope of Iteration V0.8 is defined strictly by the goal of implementing the V0.7 theoretical models into a functional, multi-agent, multi-AI coding collaboration protocol engine. The scope encompasses the following defined boundaries and exclusions.

## 1. In Scope
- **`.agent-state/` Physical Migration:** Modify the `bin/agent-protocol-init` bootstrap script and the `src/mcp-server/src/index.ts` path resolution logic to operate against a target repository's `.agent-state/` directory while maintaining backward compatibility.
- **Dynamic Registry & Orchestration:** Implement the environment fingerprinting and capability modeling registry (`.agent-state/registry.json`).
- **SubAgent Review Automation:** Implement a functional background hook or CLI command (`agent-protocol-review`) that autonomously invokes an isolated LLM (e.g., via `opencode run`) to execute the Gatekeeper review protocol.
- **Legacy Tooling Deprecation:** Inject deprecation warnings into `scripts/doctor.py`, `scripts/verify_protocol.py`, etc., as defined in the V0.7-M5-B roadmap.

## 2. Out of Scope (Strict Exclusions)
- **Unapproved Rule Changes:** Changing the 6-stage lifecycle (Harness -> Acceptance) or the 5-artifact taxonomy (Contract -> Post-Gate) without formal review.
- **Removing Legacy Tools:** Physically deleting the legacy Python scripts (`scripts/*.py`). V0.8 only executes the deprecation warning phase.
- **Agentic Sycophancy:** Permitting the Executor (e.g., Hermes) to generate its own `decision.json` review artifact. The Gatekeeper SubAgent must remain mathematically and procedurally isolated.
- **Hardcoding Participants:** The system must not hardcode specific IDEs or Agents (e.g., "Must use OpenCode"). It must rely on the dynamic capability registry.
