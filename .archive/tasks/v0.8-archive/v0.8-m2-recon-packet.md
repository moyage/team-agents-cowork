# V0.8-M2 Reconnaissance: The LLM Router & Dynamic Capability Registry

## 1. Objective
Design and specify the dynamic registry (`.agent-state/registry.json`) and the LLM Workflow Router. The goal is to move away from hardcoding tool paths or agent names and instead allow the framework to dynamically assess available capabilities (e.g., `supports_mcp`, `is_isolated_reviewer`) and route incoming intents to the correct YAML workflow and target agent.

## 2. Analytical Scope

### A. Dynamic Capability Registry (`registry.json`)
- **Structure:** A JSON file residing in `.agent-state/` that acts as a service discovery layer.
- **Registration Process:** When tools are initialized or manually added, they register their profile:
  - `id`: e.g., "hermes-local", "opencode-sisyphus", "cursor-user"
  - `type`: "executor" | "reviewer" | "orchestrator"
  - `capabilities`: Array of tags like `["mcp_client", "background_process", "terminal_access"]`
  - `invocation_hook`: The command or method used to trigger this agent (if it supports programmatic triggering, e.g., `opencode run`).

### B. The LLM Workflow Router
- **Trigger:** A natural language intent from the user (e.g., "Build the login page").
- **Process:**
  1. The Router (acting via the Coordinator) reads the descriptions and `use_when` conditions of all available `.yaml` workflows in `.agent-state/workflows/`.
  2. The LLM evaluates the user's intent against these workflows.
  3. **Output:** The LLM outputs a deterministic routing decision (e.g., `{"selected_workflow": "feature-dev.yaml", "confidence": 0.95}`).
- **Task Assignment (The Link):** Once a workflow is selected, its nodes are parsed into `execution-contract.json` artifacts. The Orchestrator assigns these contracts to agents by matching the node's required capabilities against the `registry.json`.

## 3. Constraints
- **In-Scope:** Designing the schema for `registry.json` and the prompt template/logic for the LLM Router. Creating scaffolding in `src/mcp-server/src/router.ts`.
- **Out-of-Scope:** Porting the actual 17 workflows (this belongs to M3). Implementing the background review daemon (M4). Modifying active `workflow/dispatch.json` state semantics outside of standard M2 progression.
