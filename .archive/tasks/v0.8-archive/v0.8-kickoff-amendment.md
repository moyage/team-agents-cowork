# V0.8 Kickoff Amendment: The YAML DAG Engine & Archon Workflow Porting

## 1. Rationale for Amendment
The initial V0.8 Kickoff proposed a dynamic registry and a dual-tiered sync topology, but still assumed the underlying protocol would enforce the hardcoded 6-stage lifecycle established in V0.7.

A deep dive into the `Archon` open-source framework revealed a significantly more powerful orchestration paradigm: **YAML-based Directed Acyclic Graph (DAG) workflow routing.**

To achieve the "ideal end-state" for `team-agents-cowork`—a framework that supports heterogeneous agents and multi-person team domains—we must augment our architecture by adopting Archon's routing intelligence while retaining our L2/L3 zero-trust Gatekeeper defenses.

## 2. Amended Core Requirements for V0.8

This amendment injects the following massive architectural upgrades into the V0.8 implementation scope:

### A. The YAML DAG Workflow Engine
- **Old Paradigm:** A rigid, hardcoded progression through `dispatch.json` (Harness -> Workflow -> Orchestration -> Execution -> Evaluation -> Acceptance).
- **New Paradigm:** The framework will ship with a YAML parser. Users can define custom workflows in `.agent-state/workflows/*.yaml`. Workflows will support explicit dependencies (`depends_on`), conditional logic, and looping.
- **The Catch (Governance First):** Unlike Archon, which executes the LLM API directly, our YAML engine will emit `execution-contract.json` artifacts at each AI node, forcing the execution to be claimed by a registered heterogeneous tool (Cursor/Trae/OpenCode) and audited by the L2/L3 Gatekeeper.

### B. The LLM Workflow Router
- **Old Paradigm:** The user manually sets the `active_task`.
- **New Paradigm:** When a user submits a natural language intent (e.g., "Fix this bug"), the Main Brain (Coordinator) uses an LLM to evaluate all available YAML workflows and dynamically selects the most appropriate one, outputting the routing decision.

### C. Built-in Workflow Porting (The "Archon 17")
- **New Requirement:** `team-agents-cowork` must ship with a robust set of built-in, high-frequency workflows out-of-the-box. We will port and adapt Archon's 17 default workflows (e.g., `assist`, `fix-github-issue`, `idea-to-pr`, `smart-pr-review`, `piv-loop`) to our federated, JSON-contract-based architecture.

### D. Custom Workflow Authoring
- **New Requirement:** Full support for custom workflow authoring, allowing teams to define bespoke pipelines tailored to their specific project requirements or personal domain scenarios.

## 3. Impact on Existing Carry-Overs
- **`.agent-state/` Physical Migration:** Remains mandatory. The state separation is even more critical now to house the dynamic YAML workflows alongside the execution history.
- **SubAgent Review Automation:** Remains mandatory. The YAML nodes for `review` or `evaluate` will rely heavily on the automated background Gatekeeper.
