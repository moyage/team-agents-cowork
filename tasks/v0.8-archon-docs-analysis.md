# Archon Documentation Analysis & Target Blueprint

This report details a deep-dive analysis of Archon's documentation structure and content strategy, based on the `packages/docs-web` source code within their reference repository. The goal is to define a high-quality structural and visual standard to emulate for the `team-agents-cowork` framework's documentation.

## 1. Documentation Architecture & Navigation

Archon uses Astro Starlight for its documentation portal. The navigation is highly structured and clearly separates the **narrative learning path** from the **encyclopedic reference path**. 

Their primary sidebar groups are:

1. **The Book of Archon (`book/`)**: A narrative, chapter-by-chapter walkthrough (e.g., "What is Archon?", "First Workflow"). It focuses on the "Why" and builds conceptual understanding progressively.
2. **Getting Started (`getting-started/`)**: Fast onboarding, installation, quick-start, and core concepts. Highly action-oriented.
3. **Guides (`guides/`)**: Deep dives into specific capabilities (e.g., "Authoring Workflows", "Authoring Commands", "Hooks").
4. **Adapters (`adapters/`)**: Platform integrations (Slack, GitHub, Web, Telegram).
5. **Deployment (`deployment/`)**: Hosting and execution environments (Docker, Local, Cloud).
6. **Reference (`reference/`)**: Encyclopedic lookups (API, CLI commands, Variables, Config, Troubleshooting).
7. **Contributing (`contributing/`)**: Internals, releasing, developer experience quirks.

## 2. Content Strategy & Tone

* **Direct and Developer-Focused**: The tone is punchy and devoid of fluff. Sentences are short. (e.g., *"Archon makes your AI coding assistant predictable. Not by limiting it — by giving it a process to follow."*)
* **"Show, Don't Tell"**: Concepts are immediately followed by concrete YAML or Bash examples.
* **Progressive Disclosure**: High-level concepts are explained first with simple analogies, then mapped to actual configuration syntax.

## 3. Visual Elements & Formatting Standards

Archon's documentation achieves high readability without relying on complex plugins. They use standard Markdown features exceptionally well:

### A. Frontmatter Routing
Every file begins with structured YAML frontmatter for searchability and ordering:
```yaml
---
title: Authoring Workflows
description: Create multi-step YAML workflows with DAG nodes.
category: guides
sidebar:
  order: 1
---
```

### B. Markdown Tables for Comparisons
Instead of long paragraphs, relationships and concepts are mapped in clean tables:
| Concept | What It Is | Think of It Like |
|---------|-----------|-----------------|
| **Command** | Instructions for AI to execute a task | A function |
| **Workflow** | YAML file orchestrating commands | A script |

### C. Text-Based Architecture Diagrams
Instead of heavy Mermaid.js graphs that might break in CLI viewers, Archon uses elegant, pre-formatted code blocks for diagrams:
```text
Layer 1: scope
Layer 2: code-review  |  security-review   (concurrent)
Layer 3: synthesize
```

### D. Annotated Code & CLI Blocks
CLI examples always include bash comments explaining the action:
```bash
# List available workflows
archon workflow list

# Run a code review
archon workflow run smart-pr-review
```
YAML configuration blocks are frequent and represent exactly what the user would type.

### E. Blockquote Callouts
Important nuances (like CLI vs Server behavior) are highlighted using standard blockquotes:
> **CLI vs Server:** The CLI reads workflow files from wherever you run it...

## 4. Blueprint for `team-agents-cowork`

To match Archon's "high quality" standard, we must restructure our `documentation/EN` and `documentation/ZH` directories to follow this exact blueprint.

### Proposed Directory Structure

```text
documentation/
├── EN/ (and ZH/)
│   ├── book/               # Narrative tutorials (The "Why" and step-by-step walkthroughs)
│   ├── getting-started/    # Installation, Quick Start, Core Concepts
│   ├── guides/             # Feature-specific deep dives (e.g., building specific agent teams)
│   ├── integrations/       # LLM Providers, external tools, platform hooks
│   ├── deployment/         # Docker, Local run, Cloud deployment
│   └── reference/          # API specs, configuration schemas, CLI commands
```

### Mandatory Content Standards for New Docs

1. **Frontmatter**: Every `.md` file MUST contain `title`, `description`, and `sidebar: order` frontmatter.
2. **Separation of Concerns**: Do not mix tutorial steps with reference material. If a user needs to know *how* to do something, it goes in `guides/`. If they need a list of all possible parameters, it goes in `reference/`.
3. **Table-First Explanations**: Use Markdown tables whenever introducing 3 or more related concepts, parameters, or choices.
4. **Textual Diagrams**: Use ASCII/text-based diagrams in code blocks for flow charts and dependency graphs to ensure 100% terminal and Git-host compatibility.
5. **Runnable Examples**: Every YAML or Bash block must be copy-pasteable and represent a realistic, working example. Use comments inside code blocks to explain non-obvious lines.
6. **Clear Headings & Rules**: Use `---` horizontal rules to create clear visual separation between major sections within a document.
## 4. Live Documentation Supplement (https://archon.diy/book/what-is-archon/)

To augment the structural analysis of the codebase, a direct extraction of the live Archon documentation portal was performed.

### The Conceptual Triumvirate
The live documentation explicitly defines Archon using three foundational ideas, using a "Recipe + Chef" analogy:
1. **Command:** A markdown file containing instructions for the AI (The "Function").
2. **Workflow:** A YAML file that orchestrates commands (The "Script" / "Recipe").
3. **Isolation:** Git worktrees providing separate execution directories (The "Sandbox").

### Translating Archon Concepts to `team-agents-cowork`
To match Archon's clarity, our documentation rebuild must frame our architecture using a parallel, easy-to-understand triumvirate:
1. **The Intent (Contract):** The `execution-contract.json`. (Our equivalent to their Command, but mathematically bounded).
2. **The Flow (YAML DAG):** The orchestration definition in `.agent-state/workflows/`. (Our equivalent to their Workflow).
3. **The Governance (L2/L3 Isolation):** The physical separation of the *Executor* (doing the work in the local bus) from the *Gatekeeper* (auditing the diff). (Our equivalent to their Worktree Isolation, which we will physically adopt in V0.9).

### Content Delivery Strategy
Our upcoming docs refactor must adopt Archon's pattern:
- Define the framework's overarching philosophy (Low Cognitive Load + Low Invasiveness).
- Immediately map those ideas to the "Contract / Workflow / Governance" concepts.
- Provide a concrete `execution-contract.json` or YAML snippet immediately following the conceptual definition (the "Show, Don't Tell" standard).
