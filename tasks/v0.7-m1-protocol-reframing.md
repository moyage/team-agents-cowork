# V0.7 Protocol Reframing: Roles & Core Boundaries

## 1. The Core Protocol vs. The Adapter Layer
To support heterogeneous AI Coding environments while maintaining strict L0-L6 governance, the framework separates into two distinct layers:
- **The Mandatory Core (Homogeneous):** `workflow/dispatch.json`, execution contracts, and independent review decisions. This is the source of truth that blocks or allows execution. It relies on a physical MCP Server (or future native abstractions) for file-driven state guarantees.
- **The Adapter Layer (Heterogeneous):** The environment-specific prompt injections (e.g., `.cursorrules`, `AGENTS.md`) and tool invocations. The adapter translates the environment's capabilities into the protocol's requirements.

## 2. Defined Roles
The V0.7 protocol recognizes four distinct agent/human roles:

### 2.1 Executor
- **Responsibility:** Implements features, writes code, or fixes bugs based on an approved `Execution Contract`.
- **Constraint:** Cannot bypass `*_contract_review_pending` or `*_result_review_pending`. Must generate a structured `execution-result.json` when finished.
- **Environment:** Any IDE or CLI (Cursor, Trae, OpenCode).

### 2.2 Reviewer (Gatekeeper)
- **Responsibility:** Audits the Executor's proposed contracts and resulting Git diffs. The Gatekeeper function enforces the hard physical block until the `decision.json` artifact is Approved.
- **Constraint:** Must be structurally isolated from the Executor (e.g., a separate LLM process, a different context window, or a human). Cannot mutate source code directly. Outputs `decision.json`.
- **Environment:** A trusted API call, a CLI Sub-Agent, or a human PR reviewer.

### 2.3 Coordinator (Orchestrator)
- **Responsibility:** Reads `dispatch.json`, determines the active task, and decomposes epics into bounded sub-tasks. Assigns work to Executors.
- **Constraint:** Operates at the meta-workflow level. Cannot self-approve implementation. 
- **Environment:** Advanced background agents (OMOC, Superpowers) or a Lead Developer.

### 2.4 Evaluator (Optional/Future)
- **Responsibility:** Runs integration tests or performance benchmarks post-execution.

## 3. The Core Loop
1. Coordinator plans -> Updates `dispatch.json` (Kickoff).
2. Executor drafts Contract -> `dispatch.json` becomes `contract_review_pending`.
3. Reviewer signs off -> `decision: Approved`.
4. Executor writes code -> Generates Result Artifact -> `dispatch.json` becomes `result_review_pending`.
5. Reviewer signs off -> `decision: Approved`.
6. Coordinator marks task `Completed`.
