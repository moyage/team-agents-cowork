# V0.7 Environment Adaptation Problem Statement

## 1. Problem Context
In earlier iterations (V0.1 to V0.6), the `team-agents-cowork` framework assumed a relatively homogeneous integration environment, primarily focusing on injecting static `.cursorrules` or `CLAUDE.md` files. 

However, the reality of modern AI Coding is highly heterogeneous. Users employ various environments, including:
- **IDEs:** Cursor, Trae, Windsurf
- **CLIs & Terminals:** Claude Code, OpenCode, OMOC
- **Frameworks:** Superpowers, Openspec

These environments differ wildly in their capabilities:
- Some support native MCP (Model Context Protocol).
- Some support background processes, shell execution, or file reading tools out-of-the-box.
- Some rely solely on system prompts, while others can dynamically inject instructions based on project state.

## 2. Core Challenges
1. **Environment Fingerprinting:** The framework lacks a robust mechanism to reliably detect the active AI Coding environment dynamically. It relies on explicit user declaration (e.g., `setup.sh --ide cursor`).
2. **Capability Variability:** If an agent operating via Claude Code attempts to use a tool only available in OpenCode, the workflow will fail. The framework does not currently map or enforce environment-specific capabilities.
3. **Adapter Drift:** As AI Coding tools rapidly evolve, their prompt injection mechanisms and supported schema formats change. The framework's static adapters (`adapters/*.tpl`) are prone to drift and obsolescence.
4. **Non-Intrusive Integration:** Forcing users to adopt a single toolchain (e.g., "You must use OpenCode") violates the framework's core principle of "Low Invasiveness."

## 3. Desired Outcomes
To support "Low Invasiveness" and "Low Cognitive Load," V0.7 must solve the adaptation problem by building:
- **Environment Discovery:** A standard protocol for identifying the active environment.
- **Capability Modeling:** A registry defining what tools and contexts are available in a given environment.
- **Compatibility Matrix & Drift Detection:** Mechanisms to track adapter versions against known environment updates and detect when an adapter is failing or outdated.
- **Adapter Evolution:** The ability for the framework to automatically generate, propose, and (upon approval) implement bounded fixes to adapters when drift is detected.
