# Code Review Report: team-agents-cowork (v0.1.0-draft → v0.8.5)

**Review Date:** 2026-04-17  
**Scope:** Full codebase read-only audit across `src/`, `bin/`, `scripts/`, `templates/`, `documentation/`, and configuration files.  
**Objective:** Identify Architecture Drift, Redundant Code, Hardcoding, Security Vulnerabilities, and Documentation Mismatches.

---

## 1. Architecture & Protocol Drift

### 1.1 Hardcoded Agent Names in Registry Scaffold
- **File:** `bin/agent-protocol-init:50-61`
- **Issue:** The default `registry.json` hardcodes agent IDs `"hermes-local"` (type: `orchestrator`) and `"opencode-sisyphus"` (type: `executor`). This directly contradicts the V0.8 M2 Dynamic Participant Registry principle, which states agents must register capabilities dynamically rather than being scaffolded with static names by the Main Brain.
- **Impact:** New projects are seeded with outdated, vendor-specific agent bindings instead of a capability-agnostic registry.

### 1.2 Mocked L3 Gatekeeper in Review Daemon
- **File:** `bin/agent-protocol-review:90-125`
- **Issue:** The `agent-protocol-review` daemon hardcodes `reviewerAgentId = "mock-isolated-reviewer"` and spawns a fake SubAgent via `execSync(`node -e "..."`)` to write an auto-approved decision JSON. Line 108 even admits: *"In a real production scenario, this would execute: execSync(`opencode-bridge run <task> --verify="<verify_command>"`")"*.
- **Impact:** The Blackbox L3 separation of duties exists only on paper. The review daemon is a simulation, not a functional isolation boundary.

### 1.3 Hardcoded Capability Requirements in Workflow Templates
- **Files:**
  - `templates/workflows/architect.yaml:9` → `capability_requirements: ["opencode-core-only"]`
  - `templates/workflows/architect.yaml:15` → `capability_requirements: ["cursor-human-in-loop"]`
  - `templates/workflows/custom-deploy-pipeline.yaml:21` → `capability_requirements: ["cursor-human-in-loop"]`
  - `templates/workflows/idea-to-pr.yaml:9` → `capability_requirements: ["architect-core"]`
- **Issue:** These workflows bind to specific agent implementations/brands (`opencode`, `cursor`) rather than abstract capabilities (e.g., `human_in_loop`, `architecture_review`). This violates the framework's stated goal of routing by capability rather than hardcoding agent names.

### 1.4 Clean Areas
- **No `is_isolated_reviewer` bindings** found in `src/` or `bin/`.
- **No naked `opencode run` CLI calls** found in active source code (only appears in archived task files and comments).

## 4. Security Vulnerabilities

### 4.1 Shell Injection in Review Daemon
- **File:** `bin/agent-protocol-review:119`
- **Code:**
  ```js
  const command = `node -e "const fs=require('fs'); fs.writeFileSync('${artifactFullPath}', JSON.stringify({...}), 'utf8');"`;
  execSync(command, { stdio: 'inherit' });
  ```
- **Issue:** `artifactFullPath` is interpolated directly into a shell command string. If the path contains a single quote (`'`) or shell metacharacters, an attacker could inject arbitrary shell commands.
- **Severity:** Medium. Requires control over `dispatch.json` contents, but the framework is explicitly designed to allow AI agents to modify state files.
- **Correction to Previous Report:** The claim that `src/mcp-server/src/index.ts` has OS command injection via `execAsync(\`git diff --name-only ${baseline_commit}\`)` is **overstated**. While `baseline_commit` is passed into a template string, `execAsync` (promisified `exec`) with a `git diff` command is a shell invocation. However, the MCP server runs with the privileges of the local AI coding tool, making this an internal trust-boundary issue rather than a remotely exploitable vulnerability. The more acute injection risk is in `bin/agent-protocol-review` where `execSync` is used with an inline `node -e` script.

### 4.2 Path Traversal in Review Daemon
- **File:** `bin/agent-protocol-review:79-114`
- **Issue:** The `expectedArtifact` value is read directly from `dispatch.json` and passed into `path.join(PROJECT_ROOT, expectedArtifact)` without sanitization. If `expectedArtifact` contains `../` segments, `fs.mkdirSync(artifactDir, { recursive: true })` will create directories **outside** the project boundary, and the decision file will be written outside `PROJECT_ROOT`.
- **Severity:** Medium. Could lead to file system pollution or writing into sensitive parent directories.
- **Correction to Previous Report:** The claim that `verify_execution_compliance` in `src/mcp-server/src/index.ts` allows arbitrary file reads via `contract_path` is **partially overstated** — while `contract_path` is resolved with `path.resolve`, there is no `startsWith` boundary check on the contract file (unlike the robust checks on `cowork_read_file`). This is a valid inconsistency but is lower severity than claimed because the MCP server is a local-only stdio transport.

### 4.3 Security Boundaries Properly Implemented
- **MCP Server (`src/mcp-server/src/index.ts`):** Path traversal is correctly mitigated in `cowork_read_file`, `cowork_write_file`, and `cowork_list_directory` via `startsWith(resolvedBase)` checks.
- **No `eval()` calls** were found anywhere in the active codebase (`src/`, `bin/`, `scripts/`).
- **Worktree Manager (`src/mcp-server/src/worktree-manager.ts`):** Git commands are invoked via `execFile` with discrete argument arrays, eliminating shell injection risks.

## 5. Documentation Mismatches

### 5.1 Incorrect Workflow Count
- **Files:** `README.md:38`, `README_zh-CN.md:38`, `documentation/EN/reference/built-in-workflows.md`, `documentation/ZH/reference/built-in-workflows.md`
- **Issue:** Documentation claims there are **17** built-in workflows, but `templates/workflows/` actually contains **18** YAML files (`custom-deploy-pipeline.yaml` is present in the directory but omitted from the built-in table).

### 5.2 Inflated Reviewer Count
- **File:** `templates/workflows/comprehensive-pr-review.yaml`
- **Issue:** The description claims *"Multi-agent PR review (5 parallel reviewers)"*, yet the YAML only defines **2** review nodes (`syntax_review`, `logic_review`) plus a synthesis node.

### 5.3 Missing Referenced File
- **Files:** `adapters/AGENTS.md.tpl:11`, `adapters/CLAUDE_CODE_RULES.tpl:5`
- **Issue:** Both templates reference `documentation/EN/SYSTEM_PROMPT.md`, which **does not exist** in the repository.

### 5.4 Non-Existent Bridge Tool Documented as Standard
- **Files:** `documentation/EN/core-concepts/governance-protocols.md:20`, `documentation/ZH/core-concepts/governance-protocols.md:20`
- **Issue:** Governance documentation mandates that the Orchestrator *"MUST use `opencode-bridge run <task> --verify="<verify_command>"`"*. No such `opencode-bridge` binary or wrapper exists in the codebase. The actual `bin/agent-protocol-review` only contains a mocked inline Node.js script.

### 5.5 Stale Schema Reference
- **File:** `documentation/EN/reference/schema-reference.md`
- **Issue:** The `Execution Contract Schema` block shown in the documentation is missing the `delegation_mode` and `capability_requirements` fields that were added in V0.8 and are present in the canonical `schemas/execution-contract.schema.json`.

### 5.6 EN/ZH Parity
- **Result:** English and Chinese documentation are structurally synchronized and consistent in content. No bilingual drift was detected.

## Summary of Recommendations (Read-Only)

1. **Architecture:** Replace hardcoded agent names in `bin/agent-protocol-init` with a minimal, capability-only template. Upgrade `bin/agent-protocol-review` from a mock to a real SubAgent bridge or clearly mark it as `TODO/Mock` in the CLI output.
2. **Cleanup:** Remove the three deprecated Python scripts, delete the stub `render_adapters.py`, fix the broken `Makefile` `sync` target, deduplicate `CHANGELOG.md`, and consolidate uninstall logic into a single entry point.
3. **Hardcoding:** Replace the absolute path in `scripts/verify_mcp_refactor.sh` with a relative path or `$(dirname "$0")`. Reconcile the five divergent version strings.
4. **Security:** Refactor `bin/agent-protocol-review` to use `fs.writeFileSync` directly instead of `execSync` with string interpolation, and validate/sanitize `expectedArtifact` against path traversal before calling `fs.mkdirSync`. Consider adding a `path.startsWith` boundary check for `contract_path` in `verify_execution_compliance` to match the other file-tool handlers.
5. **Documentation:** Update workflow counts to 18, fix the `comprehensive-pr-review.yaml` description or add the missing 3 review nodes, create the missing `SYSTEM_PROMPT.md` or remove the references, and update the schema reference to include `delegation_mode` and `capability_requirements`.
