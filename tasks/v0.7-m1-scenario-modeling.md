# V0.7 Scenario Modeling: Personal Mode & Team Mode

## 1. Goal
Model how the V0.7 Core Collaboration Protocol operates under different team scales and heterogeneous AI coding combinations without introducing prohibitive friction.

## 2. Personal Mode (Single Developer + Single AI Environment)
**Scenario Description:** A lone developer using a primary AI Coding tool (e.g., Cursor) to implement features rapidly.
**The Problem:** Generating a formal JSON contract and manually waiting for an independent review introduces high cognitive load and context switching for a single person.
**The V0.7 Personal Mode Model:**
- **Roles:** The Developer acts as the Coordinator and (optionally) the Gatekeeper. The AI acts as the Executor.
- **Workflow Optimization:** 
  - The AI drafts the `Execution Contract`.
  - To maintain the "Governed Autonomy" principle without losing the physical `decision.json` artifact, the framework offers two paths:
    1. **Automated Sub-Agent (Low Friction):** An automated background script (or MCP tool, e.g., `agent-protocol-review`) triggers a separate LLM call (e.g., GPT-5.4) to instantly review the contract and generate the `Approved` JSON.
    2. **Human-in-the-Loop (Fallback):** The developer manually reviews the `tasks/*-execution-contract.json` and runs a quick confirmation script to sign it off.
- **Heterogeneity:** The single AI tool (e.g., Cursor) can leverage native MCP via `.cursorrules`.

## 3. Team Mode (Multi-Developer + Multi-Agent Orchestration)
**Scenario Description:** A team where Dev A uses Trae, Dev B uses Cursor, and a background agent (OMOC/Superpowers) manages the project state.
**The Problem:** Ensuring different tools, with different capabilities, don't overwrite each other's state or bypass architectural guidelines.
**The V0.7 Team Mode Model:**
- **Roles:**
  - **Coordinator:** A dedicated Orchestrator Agent (e.g., running via OpenCode in the background) reads `dispatch.json` and decomposes a Jira epic into 5 JSON task contracts.
  - **Executors:** Dev A (Trae) picks up Task 1. Dev B (Cursor) picks up Task 2. Their local adapters ensure their IDEs read only their assigned JSON contract.
  - **Gatekeepers:** An automated CI pipeline (e.g., GitHub Actions) or a dedicated Reviewer Agent (running locally or remotely) audits the generated `tasks/*-execution-result.json` and the corresponding Git Diff before allowing a PR merge.
- **State Synchronization:** The `workflow/` and `tasks/` directories remain the shared source-of-truth. All agents and humans read from and write metadata to these directories, preventing conflicts. (Note: Extraction to `.agent-state/` is deferred to later milestones).
- **Heterogeneity:** The protocol standardizes the *hand-off* (JSON artifacts, Git Diffs) rather than the *implementation*. Trae and Cursor can both generate compliant JSON files, even if their internal prompts differ wildly.

## 4. Environment Adaptation Constraints
- **Low Invasiveness:** Neither mode forces the user to switch their primary IDE. The framework must rely on standardized data contracts (`dispatch.json`, `execution-contract.json`) that any sufficiently advanced AI can read and write.
