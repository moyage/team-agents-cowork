# V0.8-M1 Reconnaissance Packet: The Agent Workspace & Core DAG Parser

## 1. Objective
To design the `.agent-state` migration and define a new YAML-based Directed Acyclic Graph (DAG) workflow schema ("Archon-lite" Engine). This replaces the legacy hardcoded 6-stage lifecycle, introducing a dynamically parsed DAG structure where execution transitions through our 5-Artifact JSON taxonomy with Gatekeeper approvals.

## 2. New `.agent-state/` Physical Directory Structure
The `.agent-state/` directory replaces the legacy workflow tracking mechanisms.

```text
PROJECT_ROOT/
└── .agent-state/
    ├── workflows/
    │   └── *.yaml                  # DAG workflow definitions (e.g., build-feature.yaml)
    ├── tasks/
    │   ├── execution-contract.json # L2 Intent (Generated when node execution starts)
    │   ├── pre-gate.json           # Pending Gatekeeper approval to execute
    │   ├── execution-result.json   # Output from the node execution
    │   └── post-gate.json          # L3 Post-Gate approval (Allowing DAG to proceed)
    └── workflow/
        └── dispatch.json           # Current workflow dispatch state / execution pointer
```

**Backward Compatibility Logic:**
In `src/mcp-server/src/index.ts`, path resolution for dispatch logic will be updated to:
1. Prefer `PROJECT_ROOT/.agent-state/workflow/dispatch.json`
2. Fallback to `PROJECT_ROOT/workflow/dispatch.json` for legacy compatibility.

## 3. YAML Schema for Workflows (The "Archon-lite" Engine)
Workflows are defined as a list of nodes, where each node specifies its dependencies via `depends_on`. The engine guarantees nodes only execute once their dependencies complete the full artifact lifecycle.

### Concrete Example: `build-feature.yaml`
```yaml
name: build-feature
version: 1.0
description: Standard feature implementation workflow
nodes:
  - id: analyze-requirements
    description: Parse user request and determine the files that need modification.
    depends_on: []

  - id: design-architecture
    description: Draft the technical design and architectural changes needed.
    depends_on: 
      - analyze-requirements

  - id: write-code
    description: Implement the feature based on the approved technical design.
    depends_on:
      - design-architecture

  - id: run-tests
    description: Execute unit and integration tests against the new code.
    depends_on:
      - write-code
```

## 4. DAG Parsing Logic and the 5-Artifact JSON Taxonomy
The DAG engine does not call LLM APIs directly. Instead, processing a node triggers the following artifact lifecycle:

1. **Contract Generation:** The engine reads the node's `description` (prompt) and generates an `execution-contract.json` (L2 Intent) in `.agent-state/tasks/`.
2. **Pre-Gate Wait:** The system creates a `pre-gate.json` (or uses the contract as the pre-gate) and halts execution, waiting for the Gatekeeper (user or higher-level agent) to Approve.
3. **Execution:** Once approved, the execution occurs autonomously.
4. **Result Generation:** The execution output is written to `execution-result.json`.
5. **Post-Gate Wait:** The engine pauses again, waiting for an L3 `post-gate.json` approval verifying the execution succeeded.
6. **DAG Progression:** Only upon Post-Gate approval does the engine mark the Node as complete, allowing downstream nodes (defined in `depends_on`) to begin their own artifact lifecycle.

## 5. Constraints (Out-of-Scope Items)
- **Automatic Retries:** If a Gatekeeper rejects a pre-gate or post-gate, retry logic or auto-reverting is out of scope for V0.8-M1.
- **Dynamic Node Generation:** Workflows must be statically defined in YAML. Dynamically adding new nodes to the DAG during execution is not supported in this iteration.
- **Cross-Workflow Dependencies:** Nodes can only depend on other nodes within the same `.yaml` file.
- **Complex Conditional Logic:** Branching or `if/else` logic within the YAML DAG is omitted; all paths follow standard `depends_on` fulfillment.
