# V0.8 Planning Q&A: Orchestration, Registry, and Workspace Topologies

## Question 1: Participant Registration in the Personal Domain
**Q: In the `team-agents-cowork` framework, should the "participants" in the personal domain (Hermes, SubAgents, OpenClaw, Opencode suite) be hardcoded/selected by the Main Brain, or should there be a dynamic registration/deregistration mechanism throughout the lifecycle?**

**A: A Dynamic, Capability-Based Registry is Mandatory.**
Hardcoding the participants violates the core principle of "Low Invasiveness" and limits the system's ability to evolve. The Main Brain (Hermes) should not assume `Opencode` or a specific SubAgent is always available or the best tool for every sub-task.

- **The Mechanism (Dynamic Registry):** The framework must introduce an `Environment & Capability Registry` (as conceptualized in V0.7-M4). When a new tool (e.g., `Openspec` or a new `Momus` reviewer) comes online in the personal domain, it "registers" its capabilities (e.g., `supports_mcp`, `can_run_tests`, `is_isolated_reviewer`) into the local `.agent-state/registry.json`.
- **Lifecycle Action:** When the Main Brain (acting as Coordinator) breaks down an Epic into `execution-contract.json` files, it assigns tasks based on *required capabilities*, not hardcoded names. If `Opencode` is busy or deregisters, the Main Brain dynamically routes the task to the next available agent with matching capabilities (e.g., prompting the human to use Cursor).

## Question 2: Team Mode Role Assignment & Orchestration
**Q: If a complex project is managed in Team Mode, how should different team roles "participate", be "assigned", and "orchestrated" regarding Question 1?**

**A: Decentralized Claiming via the State Machine (The "Pull" Model).**
In a team environment, a centralized "Main Brain" cannot dictate the local environments of remote developers. Therefore, orchestration shifts from a "Push" model to a "Pull" model.

- **Participation:** Team members (human developers and their respective local AI tools) do not need to "register" with a central server. They simply clone the repository and sync the `.agent-state/` directory.
- **Role Assignment:** The central `dispatch.json` and the pool of `tasks/*-execution-contract.json` act as a job board. Contracts are marked as `PENDING`. A remote developer's local Orchestrator (or the developer themselves) reviews the pool and *claims* a contract by updating the assignee field and pushing the state change.
- **Orchestration:** The framework strictly governs the *boundaries* (via `allowed_files` in the Pre-Gate) rather than the *workers*. If Alice (using Trae) and Bob (using OpenCode) claim different tasks, the L2 Gatekeeper ensures their contracts do not overlap. The orchestration is implicitly managed by the conflict-free state machine.

## Question 3: Workspace Topology in the Personal Domain (V0.8 Scenario)
**Q: In the current V0.8 scenario involving only local personal domain participants, should the collaborative workspace be in the local project directory, a framework-defined shared directory, or the cloud (e.g., GitHub)?**

**A: The Local Project Directory (`.agent-state/`).**
For a purely personal domain execution, pushing state changes to a remote cloud repository for every micro-transition (e.g., `contract_review_pending` -> `execution_ready`) introduces severe latency and pollutes the Git history with orchestration noise.

- **The Workspace:** All collaboration must happen within the target project's local directory, specifically inside the newly proposed `.agent-state/` folder (the primary V0.8 architectural migration).
- **The Flow:** Hermes, Opencode, and the Reviewer SubAgents all read and write to this local `.agent-state/` folder. It acts as an extremely fast, high-bandwidth local memory bus. The results are only committed to the actual Git tree and pushed to GitHub when a task reaches the final `Acceptance` stage.

## Question 4: Workspace Topology with Remote Team Members
**Q: If Question 3 involved remote colleagues participating from their own personal domains, how should the collaborative workspace be configured?**

**A: A Dual-Tiered Sync Topology (Local Bus + Cloud Ledger).**

When remote colleagues join, the local `.agent-state/` folder can no longer function in isolation, but pushing every micro-state to GitHub `main` is still too noisy.

- **The Configuration:** 
  1. **The Cloud Ledger (GitHub):** The remote repository acts as the ultimate Source of Truth for *Milestone-Level* and *Gate-Level* transitions. A dedicated branch (e.g., `agent-sync` or tracking `.agent-state/` commits) is used to sync `dispatch.json` and the JSON contracts/decisions.
  2. **The Local Bus (Target Project):** Remote developers pull the `.agent-state/` updates. Their local AI tools (e.g., Cursor) read the contracts, execute the work locally, and invoke their local SubAgents for the L2/L3 reviews.
- **The Synchronization:** Only formally approved state transitions (e.g., an Approved `execution-result.json` accompanied by the actual code commit) are pushed back to the Cloud Ledger. This ensures that the remote team stays synchronized on *intent* and *completed work* without broadcasting the internal trial-and-error loops of their local AI tools.
