# V0.6 M4-M5 Continuity Note

## 1. Context & Rationale
In Iteration V0.6, milestone **M4 (Bootstrap & Enablement)** successfully introduced programmatic installation scripts (`bin/agent-protocol-init`). During its validation, the integration test client (`src/mcp-server/src/test-client.ts`) reliably exposed a fundamental flaw in the MCP context APIs: path resolution fragility (`ENOENT`).

## 2. Execution Relationship
Rather than bundling the API path fix into the M4 bootstrap work (which would conflate script generation with MCP server logic modifications and violate strict boundaries), the workflow naturally advanced to **M5 (Tooling Deprecation & Read-Only API Hardening)**. 

- **M4's Role:** Acted as the catalyst and diagnostic probe. By connecting to the server via stdio as an external consumer, it proved that the server's hardcoded `__dirname` paths failed when invoked from differing execution roots.
- **M5's Role:** Acted as the dedicated, bounded restoration phase. It hardened `get_dispatch_state`, `get_iteration_state`, and `get_execution_diff` to resolve against `process.env.PROJECT_ROOT || process.cwd()`, ensuring the server can support the diverse initialization patterns introduced by M4.

## 3. Governance Conclusion
This separation ensures that architectural discovery (M4 client testing) and architectural remediation (M5 server patching) maintain their own independent Execution Contracts, Result Reviews, and Artifact Trails, adhering strictly to the L2/L3 governance model.
