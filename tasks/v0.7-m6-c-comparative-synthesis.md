# V0.7-M6-C: Comparative Synthesis & V0.8 Carry-Over Extraction

## 1. Executive Summary
This document synthesizes the findings from the V0.7 Protocol Design and Pilot Validations (M6-A and M6-B). It compares the "Unified Model" (the L2/L3 Gatekeeper protocol developed in this repository) against the "Non-Intrusive Schema Injection" approach (exemplified by `labs-teamwork-spec`), and formally extracts the design blueprints created in M5 as actionable epics for Iteration V0.8.

## 2. Comparative Synthesis: The Unified Model vs. Schema Injection

### The "Unified Model" (team-agents-cowork)
- **Mechanism:** Enforces collaboration through a strict, file-based state machine (`workflow/dispatch.json`) and a 5-artifact lifecycle (Contract, Pre-Gate, Result, Post-Gate). Uses a physical MCP Server to block unapproved commits.
- **Strengths:** 
  - **Absolute Auditability:** Every intention and execution is mathematically verified by an independent Gatekeeper before state progression.
  - **Eliminates Agentic Sycophancy:** AI cannot "rubber-stamp" its own work because the Gatekeeper evaluates the git diff, not just the LLM's claims.
  - **Heterogeneous Support:** As proven in the M6-B pilot, capability modeling allows different IDEs (Cursor, Trae) and CLI agents (OpenCode) to seamlessly share a project by adhering to the universal JSON handoff format.
- **Weaknesses:** High cognitive load if the JSON artifacts must be manually written by humans. (Mitigated by the V0.7 Personal Mode design where the AI assumes this burden).

### The "Non-Intrusive Schema Injection" (`labs-teamwork-spec`)
- **Mechanism:** Uses a CLI to inject rules (e.g., `.cursorrules`, `CLAUDE.md`) into a target repository, relying entirely on the native AI environment to follow those system prompts.
- **Strengths:** Zero friction. Instant onboarding. Perfectly native IDE experience.
- **Weaknesses:** Vulnerable to context window degradation and LLM hallucination. There is no physical "lock" preventing a rogue agent from deleting critical files or ignoring the injected rules.

### The Synthesized Architecture (V0.8 Target)
The ultimate architecture combines the strengths of both:
Distribute the protocol as an NPM CLI package (like `labs-teamwork-spec`) that injects the required `.agent-state/` directories and prompt adapters into any repository. However, those injected adapters will instruct the local AI to use the `team-agents-cowork` MCP Server to physically enforce the Gatekeeper state machine. 

## 3. Evidence-Based Assessment Review
- **M6-A (Personal Mode Pilot):** Validated that a single developer can utilize the 5-artifact model without friction by delegating JSON generation to the Executor AI and delegating the Gatekeeper review to a simulated background daemon.
- **M6-B (Team Mode Pilot):** Validated that a Coordinator can successfully orchestrate tasks across disparate environments (e.g., Trae vs OpenCode) using the Capability Registry and JSON contracts as a universal synchronization layer, avoiding race conditions.

## 4. V0.8 Implementation Epics (Carry-Overs)
Based on the designs generated in V0.7-M5, the following epics are formally extracted as the implementation roadmap for V0.8:

### Epic 1: .agent-state/ Physical Extraction (From M5-A)
- **Goal:** Move the `workflow/` and `tasks/` directories into a `.agent-state/` folder injected into the user's target repository.
- **Key Task:** Modify `src/mcp-server/src/index.ts` to dynamically resolve paths using `process.env.PROJECT_ROOT/.agent-state/`, implementing the backward-compatibility fallback.

### Epic 2: Legacy Tooling Deprecation (From M5-B)
- **Goal:** Phase out redundant Python validation scripts.
- **Key Task:** Inject deprecation warnings into `scripts/doctor.py`, `scripts/verify_protocol.py`, and `scripts/validate_workflow_state.py` for one full iteration cycle, ensuring MCP parity covers all legacy edge cases.

### Epic 3: SubAgent Review Automation Daemon (From M5-C)
- **Goal:** Eliminate the need for manual terminal invocation of the Gatekeeper in Personal Mode.
- **Key Task:** Build the `agent-protocol-review` daemon/CLI hook that automatically triggers a secondary LLM container upon detecting a `_review_pending` state transition in `.agent-state/workflow/dispatch.json`.
