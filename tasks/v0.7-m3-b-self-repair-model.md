# V0.7-M3-B: Improvement Proposal Pipeline & Governed Self-Repair Model

## 1. Objective
Define the pipeline through which the framework converts failure taxonomy data (defined in M3-A) into actionable improvement proposals, bounded strictly by the L2/L3 governance model to prevent unverified self-modification and rubber-stamp reviews.

## 2. Answers to the Core 6 Questions

### Q1: What can happen automatically?
The system (acting via background Coordinator or Evaluator agents) may automatically:
1. **Ingest & Parse Logs:** Read stderr, git diff conflicts, and rejected JSON review decisions.
2. **Apply Failure Taxonomy:** Determine if an error is a simple retry or a systemic drift.
3. **Draft Improvement Proposals:** Generate a new `tasks/*-execution-contract.json` that identifies the root cause and proposes a fix.
4. **Draft Implementation Code:** If the contract is approved, the system may automatically write the patch/code in a detached workspace or branch.

### Q2: What must be independently approved?
Two explicit physical gates MUST be signed off by a Gatekeeper (a separate LLM or a Human):
1. **Contract Approval:** Before *any* implementation code is executed or written to the active source tree, the generated `tasks/*-execution-contract.json` must be approved.
2. **Result Approval:** After the fix is drafted, the resulting `git diff` against the `allowed_files` must be verified and approved via `tasks/*-result-review-decision.json` before `dispatch.json` advances to `completed`.

### Q3: How to prevent silent self-modification?
- **The Physical Lock:** The MCP Server physically blocks execution validation (`verify_execution_compliance`) unless an approved JSON contract exists. 
- **The Source-of-Truth Constraint:** The active workflow state (`dispatch.json`) cannot be mutated by implementation code. If a self-repair agent tries to modify `dispatch.json` to skip the `review_pending` gate, the resulting git diff will include `dispatch.json`, instantly triggering a rejection by the independent Reviewer Gatekeeper.

### Q4: How to ensure auditability?
Every step of the self-repair loop generates a physical, timestamped artifact in the `tasks/` directory:
- The initial failure is recorded in the previous task's `result-review-decision.json`.
- The escalation is recorded in the new improvement `execution-contract.json`.
- The fix is recorded in `execution-result.json` and the final `result-review-decision.json`.
There are no "in-memory" or "chat-history-only" state transitions.

### Q5: How to prevent rubber-stamp review (Agentic Sycophancy)?
- **Algorithmic Isolation:** The Gatekeeper sub-agent must be explicitly prompted as an adversarial auditor. It should ideally be instantiated on a different LLM provider or model family (e.g., Executor = Claude 3.5, Gatekeeper = GPT-5.4).
- **Strict Schema Checks:** The Gatekeeper does not evaluate "is this code good." It evaluates "does `git diff --name-only` strictly match `allowed_files`?" and "Does the implementation include forbidden mutators?" Mathematical validation prevents subjective rubber-stamping.

### Q6: How to distinguish normal retry vs. problem patterns requiring escalation?
As defined in M3-A:
- **Normal Retry:** Immediate recovery attempted up to 3 times on the same phase gate (e.g., fixing a simple compilation error or adjusting a boundary constraint).
- **Escalated Problem Pattern:** Breaching the 3-retry limit, OR encountering a hard environment/adapter drift (e.g., a required CLI tool like `npm` is missing, or a path resolution throws `ENOENT`). Escalated patterns force the agent to halt the current feature epic and spin up an Improvement Epic.

## 3. Review Gating and Escalation Rules
If a Gatekeeper continuously rejects a self-repair proposal (e.g., 3 consecutive rejections on the *Improvement* contract):
- **Escalation to Human:** The system halts all automated background processing, sets `current_gate: blocked_requires_human_intervention` in `dispatch.json`, and outputs an alert to the user. Automated self-repair cannot loop infinitely.
