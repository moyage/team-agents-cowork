# V0.7-M5-A Reconnaissance: .agent-state/ Separation Roadmap

## 1. Objective
Design the roadmap and architectural strategy for physically separating the active workflow state (`workflow/` and `tasks/`) from the framework's codebase. The target is an independent `.agent-state/` directory housed within the user's project, establishing the framework strictly as an engine.

## 2. Analytical Scope
- **Target State:** The framework acts as an NPM package / global CLI. The user project tracks its own `.agent-state/workflow/dispatch.json` and `.agent-state/tasks/`.
- **Intermediate & Compatibility State:** The `agent-protocol-init` script will scaffold `.agent-state/` locally, but `.gitignore` will prevent tracking if run inside the framework repo (for self-testing). For backward compatibility, the MCP server will check `PROJECT_ROOT/.agent-state/workflow/dispatch.json` first, falling back to `PROJECT_ROOT/workflow/dispatch.json` if `.agent-state` does not exist.
- **Rollback Path:** Reverting the `agent-protocol-init` mapping logic and rolling back the MCP server path resolution checks.

## 3. Constraints
- **In-Scope:** Drafting the roadmap design document (`tasks/v0.7-m5-a-state-separation-roadmap.md`).
- **Out-of-Scope:** Executing any actual `.agent-state/` physical migration or MCP server logic changes. Modifying active `dispatch.json` state semantics.
