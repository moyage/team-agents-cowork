# V0.6 Risk Register

This document tracks known risks, dependencies, and blockers for the V0.6 Iteration.

## Technical Risks
- **Path Resolution Breakage (High):** Modifying `get_iteration_state` and `get_execution_diff` relative paths (`../../`) could break local integration tests or remote deployments if the git repository root is assumed incorrectly.
- **Workflow State Corruption (High):** As we plan to reorganize the `tasks/` and `workflow/` directories (M5), the MCP Server or Orchestrator scripts may lose access to the Canonical Source-of-Truth if paths aren't updated synchronously.
- **Bootstrapping Conflict (Medium):** The new install scripts (M6) might conflict with users' existing `.cursorrules` or native IDE settings if not handled defensively.

## Governance Risks
- **"Big Bang" Consolidation (High):** The Repository Normalization (M5) is inherently a large-scale file operation. If executed as a single massive commit without bounded checks, it could obscure architectural drift or accidentally delete critical schema files.
- **Auto-Approval Slippage (Medium):** The carry-over for "Approval Gating Automation" might tempt developers to build a self-approving loop, violating the dual-track isolation principle established in V0.5.

## Dependency Gaps & Blockers
- **`labs-teamwork-spec` Local Path:** The comparative analysis in M6 depends on the local availability of `/Users/mlabs/Programs/labs-teamwork-spec`. It exists, but any changes to that path would block M6.
- **Sub-Agent Automation:** We need to verify if the underlying orchestration layer (Hermes) can programmatically trigger a specific sub-agent command on a state transition before committing to M3's automation goals.

## Required User Inputs
- **Missing Inputs:** None currently identified. The V0.5 Post-Closure Study provides a solid foundation for M4-M6.
