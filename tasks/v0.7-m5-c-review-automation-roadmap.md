# V0.7-M5-C: SubAgent Review Automation Governance Roadmap

## 1. Objective
Design the governance boundaries and execution roadmap for automating Gatekeeper SubAgent invocations during `_review_pending` state transitions. The goal is to reduce human friction in Personal Mode without compromising the L2/L3 dual-track physical gating model.

## 2. Boundaries & Principles
- **No Self-Approval:** The primary Executor agent running the session CANNOT generate the `decision.json` artifact itself.
- **Physical Separation:** The automation script must spin up a completely isolated, secondary LLM process (e.g., a different background process, a different model, or an isolated container) to perform the review.
- **No "Rubber-Stamping":** The automated Gatekeeper must be explicitly prompted as an adversarial auditor. It cannot assume the code works; it must exclusively verify the `git diff` against the `allowed_files` in the contract.

## 3. Gating Conditions
The automated review bridge is triggered ONLY under the following exact conditions:
1. **State Trigger:** `workflow/dispatch.json` is updated by the Executor to either `contract_review_pending` or `result_review_pending`.
2. **Artifact Readiness:** The required upstream artifact (either the Execution Contract or the Execution Result) is physically written to disk.
3. **Execution Pause:** The primary Executor agent *halts* its active implementation loop, waiting for the Gatekeeper's decision artifact.

## 4. Verification Methods
How the Orchestrator/Executor verifies the automated Gatekeeper's output:
- **Mathematical Parse:** The Orchestrator reads `decision.json`. If the `decision` field is exactly `"Approved"`, execution resumes.
- **Rejection Loop Breaker:** If the `decision` field is `"Rejected"`, the Executor may attempt a retry. The Orchestrator must enforce the 3-retry limit (defined in M3-A). If 3 rejections occur on the same gate, the automation is suspended, and the task escalates to `blocked_requires_human_intervention`.

## 5. Implementation Roadmap (Deferred to V0.8+)
- **Phase 1 (CLI Tooling):** Create a standalone Node.js CLI tool (`agent-protocol-review`) that reads the current `dispatch.json` state, makes an API call to an LLM, and writes the `decision.json`.
- **Phase 2 (Background Daemon):** For Team Mode, integrate this invocation into a GitHub Action or CI pipeline that triggers on PR creation.
