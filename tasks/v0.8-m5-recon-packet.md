# V0.8-M5 Reconnaissance: Custom Workflow Authoring & Team Mode Sync

## 1. Objective
Validate the newly architected YAML DAG engine's ability to support custom user-authored workflows in the Personal Domain, and establish the concrete Git synchronization protocol (Dual-Tiered Sync) required to orchestrate these workflows across remote team members in Team Mode.

## 2. Analytical Scope

### A. Custom Workflow Authoring
- **Target:** Demonstrate that a developer can create a bespoke `.yaml` workflow in `.agent-state/workflows/` that the LLM Router can dynamically discover and parse.
- **Action:** Create a custom example workflow (e.g., `custom-deploy-pipeline.yaml`) that combines AI execution nodes with deterministic bash nodes (e.g., `npm run build`, `aws s3 sync`) to prove the extensibility of the `team-agents-cowork` engine beyond the default "Archon 17".

### B. Team Mode Sync (Dual-Tiered Topology)
- **Target:** Formalize the Git branching and state-sync strategy that allows multiple remote developers to claim and execute `execution-contract.json` artifacts without triggering merge conflicts on the local `.agent-state/` bus.
- **Action:** Document the explicit `agent-sync` branch methodology in the `TEAM_COLLABORATION.md` documentation. 
- **The Sync Protocol:**
  1. The Orchestrator generates the contracts and pushes the `.agent-state/tasks/` to the `agent-sync` branch.
  2. Developers pull `agent-sync`.
  3. A developer's local AI (e.g., Cursor) updates a contract's `assignee` field and executes the work locally.
  4. The local Gatekeeper (`agent-protocol-review`) approves the `execution-result.json`.
  5. The developer commits the *source code changes* AND the *approved JSON artifacts* to a feature branch, merging it back to `main` via standard PR. The noisy intermediate trial-and-error state transitions are deliberately NOT pushed.

## 3. Constraints
- **In-Scope:** Creating the `custom-deploy-pipeline.yaml` example. Updating the `TEAM_COLLABORATION.md` and `EXTENSION_GUIDE.md` (or creating a new `CUSTOM_WORKFLOWS.md`) to document the authoring and sync protocols.
- **Out-of-Scope:** Building a real-time WebSocket or database sync engine (the protocol relies strictly on Git). Executing the legacy Python tooling deprecation (M6). Modifying active `workflow/dispatch.json` state semantics outside of standard M5 progression.
