# V0.7-M3-A Reconnaissance: Failure Taxonomy & Ingestion Model

## 1. Objective
Design the foundational data models for the Self-Evolution Loop. This includes categorizing failures, defining how feedback is ingested, designing the retrospective artifact model, and explicitly drawing the boundary between a standard "retry" and an "improvement-case" that requires escalation.

## 2. Analytical Scope
- **Feedback Ingestion Model:** Where does the system get error data? (e.g., stderr from tools, rejection comments from Reviewers, test suite outputs).
- **Failure Taxonomy:** Classifying errors (e.g., Execution Boundary Violations, Syntax/Compilation Errors, Logic/Test Failures, Environment/Adapter Drifts).
- **Retrospective Artifact Model:** How iteration closures summarize failures and transform them into Carry-Overs.
- **Retry vs. Improvement-Case:** Defining the threshold where an agent must stop retrying (e.g., resetting a commit) and instead log a formal defect/improvement proposal.

## 3. Constraints
- **In-Scope:** Producing a design document (`tasks/v0.7-m3-a-feedback-taxonomy.md`).
- **Out-of-Scope:** Writing automation scripts to actually ingest this data. Modifying `dispatch.json` state semantics. Changing MCP server logic.
