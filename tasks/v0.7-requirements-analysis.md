# V0.7 Requirements Analysis

## 1. Collaboration Model
**Personal Mode vs. Team Mode (Minimal Core)**
- **Personal Mode:** A single developer working with one or more AI Coding tools (e.g., Opencode, OMOC) to implement a feature or solve a bug. The minimal core is the Execution Contract (defining scope and boundaries) and the Result Review (validating completion against the contract).
- **Team Mode:** Multiple developers, potentially with different AI Coding tools, working on complex projects. The minimal core extends to include Orchestration (task decomposition, assignment) and Coordinator roles (managing multi-agent workflows).
- **Unified Constraints:** Both modes rely on the foundational `workflow/dispatch.json` state machine to physically block hallucinated or unapproved execution.

**Multi-Agent Boundaries & Heterogeneity**
- **Mandatory Uniformity:** The core protocol (Execution Contract schema, Review Decision schema, Artifact formats, Phase Gating Rules) must be universally adopted to guarantee auditability and prevent Agentic Sycophancy.
- **Heterogeneous Flexibility:** The AI Coding environments themselves (Opencode vs. Superpowers vs. Openspec) and their specific toolsets (e.g., file reading, terminal commands, browser automation) can remain entirely heterogeneous.

**Low Invasiveness & Low Cognitive Load**
- The framework will NOT mandate the usage of a specific IDE, a specific CLI, or a specific set of system prompts.
- Instead, it will use lightweight adapters (`.cursorrules`, `CLAUDE.md`, etc.) to inject the *protocol awareness* into the user's preferred toolchain. The MCP Server will handle the heavy lifting of state retrieval and diff generation natively.

## 2. System Autonomy
**Governed Autonomy (What the System Can Do Automatically)**
- Automatically ingest feedback, identify failure modes (e.g., rejected contracts, compilation errors), and generate retrospective reports.
- Automatically propose improvement contracts (e.g., "Fix path fragility in get_dispatch_state").
- Automatically prepare implementation code within an approved contract boundary.

**Mandatory Gating (What Requires Review)**
- **Every execution phase must be gated by an Independent Reviewer (Gatekeeper).** The system CANNOT approve its own contracts or execution results. A mathematically isolated Sub-Agent (or Human) MUST generate the `decision.json` artifact.
- **No "Silent Self-Modification":** The system cannot mutate its own MCP Server logic, update workflow schemas, or delete legacy tooling without a formally approved contract and review decision.

**Ensuring Auditability**
- All proposed changes, execution contracts, and review decisions are physically written to disk as JSON/MD artifacts in the `tasks/` (or future `.agent-state/`) directory, creating an immutable, verifiable audit trail.

## 3. Environment Adaptation
**Discovery, Modeling, and Adapter Evolution**
- **Discovery:** The framework will utilize environment "fingerprinting" (e.g., detecting `process.env.OPENCODE_SESSION`, looking for `.cursorrules`, parsing `package.json` for OMOC dependencies) to identify the active AI Coding context.
- **Capability Modeling:** Create a capability registry mapping known environments to their supported actions (e.g., Does Opencode support MCP? Does Superpowers support background processes?).
- **Adapter Expiration:** Define compatibility matrices and drift detection logic. If an environment changes its prompt handling, the framework will detect the drift and propose an adapter update.
- **Non-Destructive Fixes:** If an adapter fails, the system will propose a bounded execution contract to patch the adapter without disrupting the user's primary workflow.

## 4. V0.6 Carry-over Roadmap
**State Separation (`.agent-state/`)**
- **Nature:** This is a phased evolutionary problem, not merely a design or implementation issue. The framework must transition from hosting its own execution history to gracefully spinning up isolated `.agent-state/` directories in target repositories while remaining backward-compatible.

**Legacy Tooling Deprecation**
- **Nature:** Do not delete. Create a deprecation map, route users to the new MCP Server, and phase out the Python scripts in a later, dedicated milestone once the MCP Server is universally adopted.

**Review Automation**
- **Nature:** Automate the invocation of the Gatekeeper Sub-Agent (e.g., triggering a background script on `_review_pending` state transitions) without diluting the governance strength. The review must still be performed by an isolated, heterogeneous LLM process.
