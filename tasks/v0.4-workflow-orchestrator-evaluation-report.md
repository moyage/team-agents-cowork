# V0.4 Workflow Orchestrator Evaluation Report

## 1. 评估背景与核心问题
V0.3 构建了完整的底层防篡改、规范模板和基于文件的轻量状态机 (`iteration-state.json`)。但目前，所有流程推进（从规划到编码、审查、返修）都强依赖**人类开发者**作为调度枢纽：通过 IDE Chat 或者 CLI 工具手动触发。这导致在个人复杂项目中容易出现流程遗漏，而在团队协作中则缺乏明确的交接（Handoff）和锁机制。

本轮评估旨在回答：
- 是否值得在 V0.4 引入 Workflow Orchestrator？
- 在个人域与团队多人协作双场景下，哪种架构更适合？
- 如何与当前基于 Git、文件、防篡改 CI 的协议底座完美融合？

## 2. 当前 V0.3 基线能力适配情况
**已适合被编排的资产**:
- `workflow/iteration-state.json`: 天然的 Orchestrator Context 注入点。
- `tasks/` 和 `retros/`: 明确的活跃上下文与归档上下文。
- `ci/integrity-manifest.json` 与 CI 脚本: 可作为编排流转时的强力 Guardrail (守卫条件)。
- `templates/`: Agent 生成输出的强 Schema 约束。

**仍然过于文本化、阻碍编排的环节**:
- 任务所有权 (Ownership) 与状态未结构化，交接依赖“口头承诺”。
- 需求池 (`demand-pool.md`) 为纯追加文本，Orchestrator 难以执行结构化分发。

## 3. 双场景需求定义

### 3.1 个人域复杂项目场景 (Solo Complex Projects)
**特点**: 单人多角色，本地优先，极速迭代。
**核心诉求**: 
- Orchestrator 应该是一个“副驾驶”，在背后静默管理 Context，自动切换当前关注的任务和阶段。
- 必须非常轻量，不能引入复杂的后端数据库或繁杂的启动配置。
- 依赖 Local 状态。

### 3.2 团队多人协作复杂项目场景 (Team Collaboration)
**特点**: 多人并发、角色分离、强审计、强防幻觉。
**核心诉求**:
- 支持明确的交接 (Handoff)、状态机锁死 (Locking) 和责任链 (Audit Trail)。
- Orchestrator 必须是中心化验证或强异步（基于 Git 事件）。
- 容忍一定延迟（如 PR 触发），但绝不容忍越权（如 Coder 强行修改 Reviewer 的打回意见）。

## 4. 候选方案分析

### 候选方案 1: Git / GitHub Actions 驱动型编排
- **定位**: 异步事件与工程调度底座。
- **机制**: 以 PR、Issue、Commit 为事件驱动，结合 GitHub Actions 触发对应的 Agent 脚本。
- **个人域**: 表现较差。太重太慢，无法满足本地一边敲代码一边微调的急速反馈。
- **团队协作**: 表现极佳。天然契合现有的 `protocol-integrity.yml`，权限控制明确，审查轨迹完美留痕。
- **推荐等级**: **Worth PoC (针对团队协作)**。

### 候选方案 2: 本地常驻守护进程 (Local Daemon / Watcher)
- **定位**: 本地轻量级自动化脚本。
- **机制**: 监听文件系统变动（如 `iteration-state.json` 或 `UPSTREAM_REVISION.md` 创建），自动唤起对应 CLI Agent 执行。
- **个人域**: 表现良好。无缝融入本地 IDE 体验。
- **团队协作**: 表现差。无法处理多人并发与中心化权限验证。
- **推荐等级**: **Backup Option**。

### 候选方案 3: MCP (Model Context Protocol) 集成编排
- **定位**: 工具与上下文服务器层。
- **机制**: 把现存的 Python 聚合脚本、验证脚本封装为 MCP Tools。Agent 在 IDE 内直接调用，并由 MCP Server 根据逻辑完成状态推进。
- **个人域**: 表现极佳。与 Cursor/Claude 深度绑定，Agent 自主操作感强。
- **团队协作**: 表现一般。MCP 目前偏向单兵作战，缺乏中心化调度流转。
- **推荐等级**: **Strong Fit (针对个人域 MVP)**。

### 候选方案 4: LangGraph (通用状态机)
- **定位**: 底层工作流与 Agent 状态机引擎。
- **机制**: 把 Agent 定义为节点，边作为状态流转。支持 Human-in-the-loop。
- **个人域**: 引入成本高，需写大量 Python 粘合代码，有点 Over-engineering。
- **团队协作**: 支持基于状态后端的持久化，适合构建复杂审计。
- **推荐等级**: **Worth PoC (如需重度定制)**。

### 候选方案 5: Bytedance deer-flow
- **定位**: 基于 LangGraph 的工业级增强封装。
- **机制**: 更高阶的抽象，内置更完善的重试、分支、路由。
- **评价**: 对比原版 LangGraph，它减少了样板代码，但生态绑定和学习成本仍然偏高。
- **推荐等级**: **Not Recommended (对于轻量 MVP 过重)**。

### 候选方案 6: CrewAI
- **定位**: 角色化多 Agent 协作框架。
- **机制**: 强烈的 Role/Task 绑定，Agent 依次传递结果。
- **评价**: 适合用来做“研究报告生成”或“一次性生成代码”，但较难融入本项目这种“人机混合、随时打断、随时修改上游文档”的长周期真实工程迭代流。
- **推荐等级**: **Not Recommended**。

### 候选方案 7: Temporal
- **定位**: 重型工业级工作流执行引擎。
- **机制**: 提供绝对可靠的长期执行、容错与恢复。
- **评价**: 对于企业级 DevOps 平台是神器，但对于个人开发者和中小团队而言运维灾难，违背了本仓库 `curl | bash` 就能用的初衷。
- **推荐等级**: **Not Recommended**。

### 候选方案 8: OpenHands (独立沙箱型 Workspace)
- **定位**: 自带沙箱的全能 Agent 运行时。
- **评价**: 它自身带有强烈的编排逻辑，如果引入，将会与本地 IDE 体验发生强烈冲突。不适合“基于本地现有项目的协作挂载”。
- **推荐等级**: **Not Recommended**。

## 5. 结论与路线建议

**是否建议引入 Workflow Orchestrator？**
**是。** 但必须**分层设计**，不能试图用一套引擎通吃个人与团队。

**场景首选与推荐分层：**
1. **个人域场景首选**: **MCP 集成型编排**。
   - 理由：将现有的 V0.3 防篡改基线和状态读取脚本封装为 MCP Server。开发者在 IDE 里只要说话，Agent 通过 MCP 自动推进状态并受限。零运维成本。
2. **团队协作场景首选**: **GitHub Actions 驱动的事件流**。
   - 理由：依赖 PR 和 Issue 触发。完美复用 V0.3 防篡改 CI。Handoff 依靠 GitHub 机制，留痕清晰，权限绝对物理隔离。

**最小边界控制 (V0.4 实施建议):**
- V0.4 绝对不要引入数据库驱动的重型引擎（如 Temporal 或强依赖 Redis 的后端）。
- V0.4 的 MVP 编排目标：**只做状态感知、校验阻断与有限的上下文投递，把高风险的写操作与版本发布留给人类 Gate。**
