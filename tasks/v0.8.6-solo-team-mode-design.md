# Solo/Team Workspace Mode Switch - Design Document

## Overview
The "Solo/Team Workspace Mode Switch" is a new feature for the `team-agents-cowork` framework, introduced in V0.8.6. It allows users to seamlessly toggle between two operational paradigms:
1. **Solo Mode (Local Concurrent)**: Execution happens locally using a single machine's resources, optimizing for fast feedback loops and direct file state mutations. Subagents run concurrently as local processes.
2. **Team Mode (Remote Isolated)**: Execution happens in isolated, containerized environments (remote or localized pods), optimizing for security, clear boundary separation, and scalable agent delegation.

## Conceptual Integration

### 1. Configuration Integration (`manifest.yaml` or `profiles/*.yaml`)
The mode will be a top-level configuration within the environment profiles.

```yaml
# profiles/default.yaml
workspace:
  mode: "solo" # options: "solo" | "team"
  # In solo mode, concurrency limits are tied to local CPU/RAM.
  max_concurrent_agents: 4
```

```yaml
# profiles/strict.yaml
workspace:
  mode: "team"
  # In team mode, isolation parameters dictate container/remote limits.
  isolation_provider: "docker"
  remote_endpoints:
    kimi-code-official: "https://agents.team/kimi"
```

### 2. CLI Parameter Override
The mode can be dynamically overridden at invocation via the CLI to allow quick local testing before remote deployment.

```bash
# Force solo mode for local fast execution
team-agents-cowork run v0.8.6-review-and-solo-team-mode.yaml --mode=solo

# Execute in fully isolated team mode
team-agents-cowork run v0.8.6-review-and-solo-team-mode.yaml --mode=team
```

### 3. Orchestrator Routing Logic
When the L2 Orchestrator reads the execution contract and DAG workflow:
- **If Mode == Solo**: The orchestrator spawns subagent processes locally, sharing the same workspace directory (`/Users/mlabs/Programs/team-agents-cowork`). It ensures concurrent file edits are managed via standard Git/FS locks.
- **If Mode == Team**: The orchestrator delegates tasks via API or remote container invocation. The `delegation_mode` (e.g., `orchestrated` vs `blackbox`) dictates whether the remote agent receives full orchestration capabilities or simple isolated IO streams. The workspace state is synced pre and post task execution.
