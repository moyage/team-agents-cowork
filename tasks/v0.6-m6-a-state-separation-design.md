# Framework vs. Execution-State Separation Design

## 1. Problem Statement
Currently, the `team-agents-cowork` repository acts as both the "engine" (MCP Server, templates, scripts) and the "car" (the active workflow state, task artifacts, and iteration history). This tight coupling causes:
- Cluttered root directories.
- Confusing onboarding for new agents (they see previous task artifacts and misinterpret them as active directives).
- Inability to cleanly "install" this framework into another project without dragging along its execution history.

## 2. Comparative Inspiration (`labs-teamwork-spec`)
The `labs-teamwork-spec` framework solves this by distributing an NPM CLI package. When run in a target directory, it injects scaffolding (e.g., `.cursorrules`) but leaves the CLI's source code hidden. It enforces rules via IDE prompts rather than a physical server.

## 3. Target Architecture for `team-agents-cowork`
We will adopt the CLI distribution model but retain the physical MCP Server enforcement:
1. **Framework Repository:** Contains `bin/`, `src/mcp-server/`, `adapters/`, and `documentation/`.
2. **Target Repository (User's Project):**
   - Running `agent-protocol-init` generates an `.agent-state/` directory.
   - `.agent-state/workflow/` houses `dispatch.json` and `iteration-state.json`.
   - `.agent-state/tasks/` houses all execution contracts and review decisions.
3. **MCP Server Integration:** The MCP Server's `PROJECT_ROOT` environment variable will map to the user's project root, and the tools will look specifically inside `PROJECT_ROOT/.agent-state/`.

## 4. Implementation Strategy (Deferred)
Given the massive impact on existing paths and the requirement for "no large scale migrations" in this phase, the actual migration is deferred to a future iteration (e.g., V0.7 or a dedicated migration epic). For M6-B, we will only perform local historical cleanup.
