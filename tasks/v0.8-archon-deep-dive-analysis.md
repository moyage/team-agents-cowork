# V0.8 Archon Deep-Dive Analysis & Synthesis

## 1. Objective
Analyze the Archon open-source project (https://github.com/coleam00/Archon) against the hypotheses proposed for the `team-agents-cowork` V0.8 iteration. Evaluate its workflow routing, YAML authoring capabilities, and its suitability as a foundational baseline for our multi-agent, multi-environment collaboration framework.

## 2. Hypothesis 1: Deeper Analysis of Archon
Archon is a powerful, TypeScript/Bun-based workflow engine designed to make AI coding deterministic. 
**Key Mechanisms:**
- **Git Worktree Isolation:** Like Docker for CI, Archon isolates every workflow run into a separate git worktree. This allows multiple AI agents to work on the same repository in parallel without colliding—a massive validation of the need for "physical isolation" in AI coding.
- **DAG Engine:** Workflows are Directed Acyclic Graphs defined in YAML. Nodes can be `prompt` (LLM execution), `bash` (deterministic scripts), `command` (sub-workflows), or `approval` (human-in-the-loop gates).

## 3. Hypothesis 2: Fixed vs. Dynamic Workflows
*Hypothesis: `team-agents-cowork` uses a fixed workflow, whereas Archon dynamically routes to 17 pre-defined workflows.*
**Validation: True.**
- **Archon's Approach:** Archon ships with diverse workflows (`archon-fix-github-issue`, `archon-plan-to-pr`, etc.). It uses an LLM-based Router (`packages/workflows/src/router.ts`). When a user types a prompt, the Router reads the descriptions and "Use when/NOT for" rules of all available workflows and outputs a single `/invoke-workflow` command.
- **Takeaway for V0.8:** `team-agents-cowork` currently relies on a monolithic `dispatch.json` state machine. To scale, we must adopt Archon's router pattern: building a registry of YAML-defined workflows and letting an Orchestrator dynamically select the right workflow based on the user's intent.

## 4. Hypothesis 3: Custom Workflow Authoring
*Hypothesis: Archon supports custom YAML workflows.*
**Validation: True.**
- **Mechanism:** Users define `.archon/workflows/*.yaml` files. The engine natively parses dependencies (`depends_on`), conditional logic (`when`), and gating. This allows developers to seamlessly mix deterministic nodes (running tests) with AI nodes (code generation).

## 5. Hypothesis 4: Is Archon the Ideal End-State?
*Hypothesis: Archon + Multi-Domain/Heterogeneous AI support = The ideal `team-agents-cowork`.*
**Validation: Partially True, but Architecturally Divergent.**

While Archon provides the ultimate blueprint for *defining* workflows, it is explicitly built as a **"Single-Developer Tool"** (as stated in its own `CLAUDE.md`). Extending it directly into our vision presents critical architectural hurdles:

1. **Centralized Engine vs. Heterogeneous Swarm:** Archon is a centralized engine that *calls* LLM APIs (`provider: claude`). It executes the prompts itself. `team-agents-cowork` is designed for heterogeneous environments—we don't want to execute the prompts centrally; we want to hand off a JSON contract to a user's *existing* local IDE (like Cursor or Trae) or a background daemon (like OMOC) to execute. 
2. **Missing Dual-Track Gating (L2/L3):** Archon has an `approval` node for human intervention, but it lacks the structural L2 (Pre-Gate Intent) and L3 (Post-Gate Evidence) zero-trust auditing that our Gatekeeper protocol enforces to prevent Agentic Sycophancy.
3. **Artifact Taxonomy:** Archon relies on loose bash scripts dropping files into an `$ARTIFACTS_DIR`. Our framework relies on mathematically verifiable JSON contracts (`execution-contract.json`, `result-review-decision.json`).

## 6. Synthesis: The V0.8 Roadmap Integration
We should not abandon `team-agents-cowork` to fork Archon. Instead, **V0.8 must absorb Archon's best paradigms into our heterogeneous architecture:**

1. **Adopt YAML DAG Workflows:** Replace our rigid, hardcoded 6-stage lifecycle with customizable `.agent-state/workflows/*.yaml` definitions.
2. **Adopt the LLM Router:** The Hermes Coordinator will act as the Router, dynamically selecting the correct YAML workflow based on the user's prompt.
3. **Retain JSON Contract Handoffs:** When a YAML node requires AI execution, instead of calling the LLM API directly (like Archon), the engine will generate an `execution-contract.json`. The user's preferred heterogeneous tool (Cursor/Opencode) will pick up the contract, execute it, and return the `execution-result.json`.
4. **Retain L2/L3 Gating:** Gatekeeper sub-agents will enforce the boundaries between nodes.
5. **Adopt Git Worktree Isolation:** Integrate Archon's brilliant Git Worktree isolation to allow multiple IDEs/Agents to execute concurrent tasks safely before merging back via the L3 Gatekeeper.
