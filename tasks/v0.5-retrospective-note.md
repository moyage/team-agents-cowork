# V0.5 M1 Practical Validation Retrospective (Partial)

## Observations on V0.4 Orchestration
- **What worked**: The strict gate-based artifacts (`dispatch.json`, `iteration-state.json`, `execution-contract.schema.json`) correctly bounded the execution scope. Schema validation (`validate_workflow_state.py`) effectively caught structural issues. Artifact-driven coordination prevents the agent from hallucinating approvals.
- **What friction was observed**: A fully autonomous batch execution loop is broken at the Independent Review Gate. Because Hermes is strictly defined as `Agent-Executor` and self-approval is forbidden, the orchestration immediately stalls waiting for human intervention. 
- **Was orchestration sufficient**: It was sufficient for safety (it successfully prevented unauthorized progression), but insufficient for full autonomous throughput.
- **Why human intervention was required**: To prevent "Agentic Sycophancy," the V0.4 protocol demands an independent reviewer. Since no orthogonal Reviewer-Agent is configured in the environment, Hermes correctly halted and escalated to a human for contract approval.

## V0.5 M1 Execution Phase Retrospective
- **What worked**: The strict `execution-contract.json` constraint and the explicit `allowed_files` list created a perfect sandbox. The Agent-Executor felt absolutely no pressure to touch adjacent components or legacy migration scripts, cleanly restricting the blast radius to `src/mcp-server/`. Validation natively handled via Node/TS tools worked flawlessly.
- **What friction remained**: Path resolution between the deeply nested `src/mcp-server/build/index.js` and the root-level `workflow/iteration-state.json` required careful runtime evaluation (`__dirname` / `import.meta.url`). A more robust injection pattern (like passing root paths via CLI arguments or `process.env`) would be more resilient for production.
- **Reviewer Lane Friction**: While the temporary isolated reviewer lane technically allowed the batch to proceed without humans, it structurally masks the fundamental limitation that an executor cannot impartially self-audit. In a multi-agent team scaleout, the workflow must enforce distinct cryptographic or process-level signatures for `Agent-Executor` vs `Agent-Reviewer` to prevent self-dealing.
