# V0.8 Multi-Angle Review & Repository Cleanup Report

## 1. Objective
This document executes a deep repository cleanup post-V0.8 closure and conducts a comprehensive multi-angle review of the iteration. It synthesizes the insights gained from the V0.8 Planning Q&A, the deep dives into **OpenHarness** and **Archon**, and evaluates the framework's evolution from four distinct stakeholder perspectives.

## 2. Repository Cleanup & Reorganization
To maintain open-source standards, the following housekeeping actions were performed on the repository:
1. **Target Artifacts Ignored:** The framework's own `agent-protocol-init` script scaffolds `.cursorrules`, `CLAUDE.md`, `AGENTS.md`, and `.agent-state/` in the target repository. Because `team-agents-cowork` is dogfooding itself, these files were inadvertently polluting the framework's Git tree. They have been removed and added to `.gitignore`.
2. **V0.8 Task Archival:** All intermediate `tasks/v0.8-*` artifacts (contracts, review decisions, and reports) have been migrated to `.archive/tasks/v0.8-archive/` to keep the active `tasks/` directory clean for the upcoming V0.9 iteration.
3. **Artifact Integrity:** The V0.8 Retrospective and this Multi-Angle Review remain in the active `tasks/` directory to bridge the context into V0.9.

## 3. Multi-Angle Review of Iteration V0.8

### A. The Architect's Perspective (System Design & Paradigms)
*Focus: Evolution from fixed state machines to dynamic DAGs.*
- **The Archon Revelation:** The deep dive into Archon fundamentally changed V0.8's trajectory. Archon proved that YAML-based Directed Acyclic Graphs (DAGs) and Git Worktrees are the ultimate solutions for making AI coding deterministic and concurrent. We successfully adopted the YAML DAG concept and ported the 17 default workflows (M3).
- **The OpenHarness Comparison:** OpenHarness acts as an infrastructure wrapper (memory, tool routing) for deploying agents. We realized that our framework should not try to be a centralized executor like OpenHarness or Archon. Instead, we must remain a **"Federated Collaboration Protocol."** We use `.agent-state/` as a shared local memory bus where heterogeneous agents (Cursor, Opencode, Trae) pull JSON contracts, execute them using their native tools, and return results.
- **Verdict:** The architectural pivot in V0.8 (M1 & M2) from a hardcoded 6-stage lifecycle to a dynamic LLM Router and Capability Registry was necessary to prevent the framework from becoming an invasive, monolithic IDE plugin.

### B. The Reviewer's (Gatekeeper) Perspective (Governance & Security)
*Focus: Defending against Agentic Sycophancy in an automated world.*
- **The Automation Double-Edged Sword:** V0.8-M4 introduced the `agent-protocol-review` daemon, allowing the Gatekeeper to run autonomously in the background. While this drastically reduces human cognitive load, it increases governance risk.
- **The Finding:** If the isolated LLM process (e.g., `opencode run -m "gpt-5.4"`) hallucinates a malformed `Approved` JSON, the daemon will blindly advance the `dispatch.json` state. 
- **The V0.9 Need:** We must introduce strict, deterministic JSON schema validation (e.g., using `ajv` or Python validators) *before* the daemon accepts the LLM's output. The Gatekeeper's math must remain infallible.

### C. The Tech Writer / Docs Agent Perspective (Clarity & Onboarding)
*Focus: Explaining complex multi-agent topologies to humans.*
- **The Challenge:** V0.8 introduced highly abstract concepts: "Dual-Tiered Sync", "Local Bus vs. Cloud Ledger", and "Dynamic Capability Registries". Explaining that "you don't run the YAML, the YAML creates a JSON contract that your Cursor reads" is structurally difficult.
- **The Solution:** The V0.8 Documentation Enrichment (M2) was critical. The documentation now reads like a mature open-source project (akin to Archon's documentation center), clearly separating the *Protocol Engine* from the *Execution Agents*. We ported the 17 Archon workflows and created explicit guides for Authoring and Extending.
- **The Verdict:** The bilingual documentation matrix successfully translates the framework's complex federated topology into digestible, open-source-grade documentation.

### D. The Team Member's Perspective (DX & Friction)
*Focus: Working in a remote, heterogeneous team without collisions.*
- **The Q&A Validation:** The V0.8 Planning Q&A highlighted the nightmare of syncing AI trial-and-error loops to GitHub `main`. 
- **The Dual-Tiered Sync Triumph:** Formalizing the `agent-sync` branch protocol (V0.8-M5) is a massive Developer Experience (DX) win. I can pull a contract, let my local AI fail 50 times in my `.agent-state/` sandbox, and only push the final, Gatekeeper-approved code and JSON artifact to the remote team. 
- **Capability Freedom:** The `registry.json` means I am not forced to install `Opencode` if I prefer `Trae`. As long as my tool can output an `execution-result.json`, the framework orchestrates me flawlessly.

## 4. Synthesis & V0.9 Mandate
V0.8 successfully laid the theoretical and scaffolding groundwork for a Heterogeneous DAG Engine. However, the system is currently "all engine, no fuel."

**The V0.9 Kickoff must focus purely on Runtime Execution:**
1. Building the actual Node.js parser that traverses the YAML DAGs and generates the `execution-contract.json` payloads in `.agent-state/tasks/`.
2. Implementing the Archon-inspired Git Worktree isolation so parallel agents can physically execute simultaneously without merge conflicts.
3. Hardening the `agent-protocol-review` daemon with strict schema validation.
