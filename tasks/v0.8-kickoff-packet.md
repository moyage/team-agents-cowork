# V0.8 Kick-off Packet: The Heterogeneous Collaboration Engine

## 1. Background & Context
Iteration V0.7 successfully elevated the framework from a structural tool into a formalized "Protocol-First Collaboration System." It established the 6-stage lifecycle (Harness -> Acceptance) and the L2/L3 dual-track gating model. However, V0.7 was purely a design and validation phase.

**Iteration V0.8 is the Implementation Epic.** It takes the blueprints drafted in V0.7 (State Separation, Environment Discovery, Review Automation) and turns them into functional code, transforming the theoretical protocol into a usable engine for the personal domain and multi-agent teams.

## 2. North Star Objective
> "Implement the V0.7 architectural designs to physically separate the execution state (`.agent-state/`), enable dynamic capability-based agent registration, and automate the Gatekeeper review loop, creating a fully functional, low-friction collaboration engine."

## 3. Core Requirements & Carry-overs (The V0.8 Demand Pool)
### A. V0.7 Carry-overs (Mandatory Implementation)
1. **`.agent-state/` Physical Migration:** Refactor the MCP Server (`src/mcp-server/src/index.ts`) and bootstrap scripts to decouple the active workflow state from the framework codebase. The engine must operate statelessly against a target repository's `.agent-state/` directory.
2. **SubAgent Review Automation Daemon:** Implement the background hook or CLI command (`agent-protocol-review`) that automatically triggers the isolated Gatekeeper (e.g., `opencode run -m "openai/gpt-5.4"`) upon detecting a `_review_pending` state transition.
3. **Legacy Tooling Deprecation:** Inject formal deprecation warnings into the legacy Python scripts (`scripts/*.py`) as defined in the V0.7 roadmap.

### B. New Capabilities (Dynamic Orchestration)
1. **Dynamic Participant Registry:** Implement the environment and capability registry (`.agent-state/registry.json`). Agents (Hermes, Opencode, SubAgents) must dynamically register their capabilities (e.g., `supports_mcp`, `is_isolated_reviewer`) rather than being hardcoded by the Main Brain.
2. **Capability-Based Routing:** The Orchestrator must assign tasks based on the dynamic registry, supporting both the "Push" model (Personal Domain) and the "Pull" model (Team Domain via distributed state sync).

## 4. Execution Discipline
This iteration operates under **Milestone-Level Autonomy** and strictly enforces the V0.7 **Interrupted-Run Resumption Rule**. All implementation must occur in bounded sub-packages with explicit Contract and Result reviews by the independent Gatekeeper.
