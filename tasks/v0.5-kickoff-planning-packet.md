# V0.5 Kickoff / Planning Packet
**Phase**: V0.5 (Native MCP Server & State Transition Tools)
**Status**: Approved & Ready for Execution

## Goal
Build and evolve a canonical, self-hosting engineering repository for the Team Agent Collaboration Protocol progressing strictly through defined phases. Having successfully locked and closed the V0.4 Workflow Orchestrator MVP via a strict repository cleanup, the objective has officially transitioned to **V0.5**. V0.5 upgrades the Solo Track to a Native Model Context Protocol (MCP) Server.

## Execution Milestones
### M1: Scaffold + Read-Only Probe (Current Focus)
- Implement base Node.js/TypeScript MCP Server scaffold using `@modelcontextprotocol/sdk`.
- Implement a single read-only probe tool: `get_iteration_state`.
- **Constraint**: Must be driven strictly by JSON execution contracts. Execution boundaries strictly confined to `src/mcp-server/` and registration artifacts.
- No modifying historical/legacy Python wrappers directly yet.

### M2: State Transition Tools (Future)
- Port state-mutating operations into MCP architecture.
- Ensure strict compliance with iteration-state validation logic.

### M3: Tombstone & Docs (Future)
- Tombstone Deprecation Strategy: Obsolete executable Python code (`scripts/validate_workflow_state.py`, etc.) will be replaced with explicit print/error statements pointing to the new TS architecture rather than silent deletion.
- Update `ONBOARDING.md` and related docs to instruct users to connect the native MCP Server.

## Constraints & Preferences
- **Terminology Purge**: The terms "MTP" and "MTP V3" are strictly deprecated globally. Use "Team Agent Collaboration Protocol" exclusively.
- **Agent/Team Collaboration Philosophy**: Heterogeneous local AI tools coexist, unified via Markdown artifacts and strict JSON Schema.
- **5-Step Strict Workflow**: Clarify -> Spec -> Plan -> Build -> Verify -> Audit & Merge -> Retro.
- **Hard Verification**: Verification must be physically independent. Reject "Agentic Sycophancy."
- **Execution Boundary**: No code execution or file modification without a predefined, validated `execution-contract.json`.

## Approval
This packet serves as the physical file-driven coordination baseline required before creating `v0.5-m1-execution-contract.json` and beginning M1 implementation.
