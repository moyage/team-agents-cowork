# V0.5 Kickoff Planning Packet: Native MCP Server Upgrade

## Background / Context
The project has successfully closed V0.4, validating the MVP workflow orchestrator. The objective now transitions to V0.5, which upgrades the Solo Track to a Native TypeScript Model Context Protocol (MCP) Server.

## Current Baseline Status
V0.4 baseline restored and structurally verified. Orphaned schemas and scripts recovered. Unauthorized execution states purged. The repository is locked in a clean, strictly compliant pre-execution state.

## V0.5 Objectives
1. Implement a canonical TypeScript-based Native MCP Server for the Team Agent Collaboration Protocol.
2. Formally deprecate and tombstone legacy executable code.
3. Validate team-agents-cowork v0.4 orchestration mechanism under strict enforcement.

## Scope
- Scaffold a standard Node.js/TypeScript MCP Server in `src/mcp-server/`.
- Implement a read-only probe (`get_iteration_state`) using `@modelcontextprotocol/sdk`.

## Non-Goals
- Full feature parity with all legacy Python scripts in M1.
- Expanding the scope to multi-agent concurrent write operations.

## Guiding Principles

### 1. Use the system to build the system
V0.5 的推进必须严格使用 team-agents-cowork v0.4 已建立的编排、workflow、contract、result、review gate 机制完成。  
本轮不仅交付 V0.5 的功能目标，也同时验证 team-agents-cowork 在真实多 Agent / 多人协作环境中的可用性、稳定性与可审计性。

### 2. File-driven coordination only
所有关键协作状态、执行边界、交付结果与审查结论必须以仓库中的物理文件为准。  
不得以聊天上下文、口头共识、隐含记忆或未落盘设定作为正式推进依据。

### 3. Contract before execution
任何实施动作必须先有明确、可审计、可验证的 execution contract。  
contract 必须先于执行创建，且应明确目标、范围、allowed files、验证命令、禁止事项与验收门槛。

### 4. Independent review gate
Executor 不得自签 review decision。  
每个关键阶段都必须经过人工或独立 Reviewer 的审查后，方可进入下一阶段。

### 5. Batch execution within approved boundaries
在明确批准的 planning 和 contract 边界内，允许 Agent 进行批处理式连续执行，减少人肉逐步转发的低效模式。  
但一旦触发边界外需求、证据不足、范围扩大、协议语义变更或阶段迁移，必须立即停止并等待审批。

### 6. No hidden coordination
不得依赖未记录的默认假设推进任务。  
所有影响执行结果的前提、依赖、阻塞、风险和判断，都必须在 planning packet、execution result、review comments 或 workflow state 中留下物理记录。

### 7. Process defects are managed inputs
本轮执行中暴露出的流程缺陷、协作摩擦、审查盲点、状态表达不足或编排问题，必须被显式记录。  
这些问题要么通过受控补充审批在本轮修正，要么进入下一轮迭代需求池，不得临场隐式改规则。

### 8. Separate delivery progress from protocol evolution
功能交付推进与协作协议演进必须区分处理。  
如发现协议本身需要升级，必须通过显式 planning / review 流程纳入，不得借实施任务顺带修改协作规则。

### 9. Workflow state must reflect reality
`workflow/iteration-state.json` 必须真实反映当前阶段、已完成事项、阻塞事项、待审事项与下一步行动。  
状态文件不得滞后于真实进展，也不得被当作“事后补记”的装饰性文件。

### 10. Retrospective is part of delivery
每个阶段完成后，除了功能结果，还必须沉淀复盘结论，包括：
- 哪些协作机制有效
- 哪些机制存在摩擦或漏洞
- 哪些问题应进入下一轮需求池
- 哪些问题需要在当前轮次做受控修正

V0.5 的成功标准不仅是功能完成，还包括：  
team-agents-cowork 的协作链条在真实迭代中被有效验证、暴露问题被结构化记录、并可作为后续版本演进输入。

## Operating Model for V0.5
- V0.5 uses a gate-based batch workflow.
- Repo files are the authoritative coordination layer.
- Executor may batch-execute only within the explicitly approved scope.
- Reviewer must be strictly independent.
- Planning approval is strictly required before any M1 execution begins.
- Execution result and review gate artifacts are mandatory before advancing to the next phase transition.
