# V0.7 Scope & Milestone Map

The scope of Iteration V0.7 encompasses the strategic elevation of the `team-agents-cowork` repository into a multi-agent, multi-AI coding collaboration protocol system. It formally integrates the findings from the V0.6 Post-Closure Normalization Study, the V0.6 Retrospective, and the V0.7 Master Kick-off Directive.

## V0.7-M1 — Protocol Reframing & Scenario Modeling
- **Objective:** Define the core protocol, multi-agent collaboration boundaries, and scenarios for Personal and Team Modes. Establish the V0.7 scope map.
- **In-Scope:** Defining the minimal core collaboration rules, role models (Executor, Reviewer, Gatekeeper, Coordinator), and boundaries for heterogeneous AI Coding environments.
- **Out-of-Scope:** Implementing code, modifying workflow state, or executing environment discovery.
- **Recommended Order:** 1

## V0.7-M2 — Harness / Workflow / Acceptance Unified Model
- **Objective:** Establish unified semantic models for harness, workflow, orchestration, execution, evaluation, and acceptance.
- **In-Scope:** Defining artifact taxonomy, task lifecycles, and review/gate/approval semantic constraints within the `workflow/` and `tasks/` paradigms.
- **Out-of-Scope:** Auto-approval logic, unreviewed state mutation.
- **Recommended Order:** 2

## V0.7-M3 — Self-Evolution Loop Design
- **Objective:** Design the governed, autonomous self-repair and evolution mechanisms.
- **In-Scope:** Designing feedback ingestion, failure taxonomy, retrospective loops, and improvement proposal pipelines, ensuring they adhere to the Phase Gating rules (Contract -> Review -> Execution).
- **Out-of-Scope:** Executing unreviewed self-modifications or bypassing the Gatekeeper.
- **Recommended Order:** 3

## V0.7-M4 — Environment Discovery & Capability Modeling
- **Objective:** Design environment fingerprinting, capability registries, and adapter evolution strategies.
- **In-Scope:** Defining compatibility/drift analysis methods and adapter abstraction rules for environments like Opencode, OMOC, Superpowers, and Openspec.
- **Out-of-Scope:** Locking the framework to a single toolchain, developing deep integrations without governed contracts.
- **Recommended Order:** 4

## V0.7-M5 — Carry-over Architecture Tracks
- **Objective:** Address V0.6 carry-overs related to state separation, legacy tooling deprecation, and review automation.
- **In-Scope:** Designing roadmaps for `.agent-state/` migration, mapping the deprecation of Python scripts (`doctor.py`, `verify_protocol.py`), and conceptualizing the automated SubAgent review bridge (`_review_pending`).
- **Out-of-Scope:** Executing the physical migration to `.agent-state/` or deleting legacy tools without explicit authorization.
- **Recommended Order:** 5

## V0.7-M6 — Pilot Validation & Comparative Synthesis
- **Objective:** Validate the new protocols via pilots in Personal Mode, Team Mode, and selected environments, synthesizing findings against `labs-teamwork-spec`.
- **In-Scope:** Conducting pilot executions, performing comparative synthesis, and generating V0.8 carry-over candidates.
- **Out-of-Scope:** Changing the other repository (`labs-teamwork-spec`), circumventing the review gating.
- **Recommended Order:** 6
