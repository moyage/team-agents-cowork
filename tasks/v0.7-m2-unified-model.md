# V0.7-M2 Unified Model: Harness, Workflow, and Acceptance

## 1. Artifact Taxonomy (The Minimal Set)
To prevent process bloat while maintaining mathematical auditability, the framework enforces a strict taxonomy of 5 JSON artifacts that drive the state machine:
1. `workflow/dispatch.json`: The **Harness/State**. Global singleton. Defines the active iteration, current gate, and blocker status.
2. `tasks/*-execution-contract.json`: The **Intent**. Defines the scope, constraints, and `allowed_files` boundary before work begins.
3. `tasks/*-contract-review-decision.json`: The **Pre-Gate**. The physical sign-off approving the contract.
4. `tasks/*-execution-result.json`: The **Evidence**. Summarizes exactly what was modified, separating code implementations from artifact tracking.
5. `tasks/*-result-review-decision.json`: The **Post-Gate**. The physical sign-off validating the git diff against the contract.

## 2. Standard Task Lifecycle
The lifecycle strictly maps to the Phase Gating model:
- `kickoff_pending` -> (Planning occurs) -> `recon_completed`
- `contract_review_pending` -> (Gatekeeper audits) -> `execution_ready`
- `execution_ready` -> (Executor implements & writes result) -> `result_review_pending`
- `result_review_pending` -> (Gatekeeper audits diff) -> `completed`

## 3. Semantic Boundaries

### 3.1 Reviewer vs. Gatekeeper
- **Reviewer** is an *action or capability* (e.g., evaluating a git diff, reading a contract, running linting checks).
- **Gatekeeper** is a *protocol function* within the workflow. A Gatekeeper enforces the hard physical block on the `dispatch.json` state machine. 
- *Clarification:* An agent acting as a Reviewer only becomes the Gatekeeper when it generates the required `decision.json` artifact that unlocks the next phase.

### 3.2 Acceptance vs. Evaluation
- **Evaluation** is the technical, often automated, assessment of the implementation. Does it compile? Do the tests pass? Does the git diff respect the `allowed_files` boundary? (Performed by the Evaluator/Reviewer).
- **Acceptance** is the human or business-level sign-off that the implementation meets the original product intent defined in the Spec/Epic. Acceptance is the final authority that closes the task.

## 4. Personal Mode vs. Team Mode (Minimal Common Core)
- **The Common Core (Must be Unified):** 
  Both modes MUST use the 5 standard JSON artifacts (Contract, Review, Execution Result) and obey the `dispatch.json` state machine. This guarantees auditability regardless of team size.
- **Local Adaptation (Allowed Differences):**
  - *Personal Mode:* The developer might use a local CLI hook to auto-trigger the Gatekeeper sub-agent, collapsing the review wait time to seconds. Acceptance is implicit when the single developer merges the branch.
  - *Team Mode:* The Gatekeeper role is fulfilled by a CI/CD pipeline blocking the PR merge. A dedicated Coordinator Agent manages task assignment, and Acceptance requires a peer or Lead Developer sign-off.

## 5. Low Cognitive Load Principle
The framework avoids process bloat by pushing the burden of JSON generation to the agents, not the humans. Humans interact via natural language (defining the epic) or high-level approvals. The agents natively generate the `contract.json`, `result.json`, and invoke the MCP Server to read `dispatch.json` without bothering the developer.
